## 🎯 エグゼクティブサマリー

cctop（Claude Code リアルタイムファイル監視システム）のv3開発ロードマップ。version-02の失敗（テスト100%成功も実環境で起動せず）から学び、**実動作駆動開発（RDD: Running-Driven Development）**を採用。6つのPhaseで段階的に機能を追加し、常に動作するシステムを維持しながら開発を進める。

## 🚀 開発方針 - RDD（実動作駆動開発）

### 従来のTDDの限界
- **テスト成功 ≠ 実動作保証**: version-02で652テスト全成功も実環境で起動せず
- **モックの罠**: モックオブジェクトが実装と乖離し、虚偽の安心感を生む
- **統合の盲点**: 個別コンポーネントは動作しても、統合時に破綻

### RDDの中核原則
1. **実行可能性最優先**: 「npm start」で即座に動作することが全ての前提
2. **日次動作確認**: 毎日必ず実環境で動作確認し、動かない変更は即座にロールバック
3. **段階的統合**: 新機能は必ず動作中のシステムに統合し、動作確認後にのみ次へ進む
4. **実ユーザー視点**: 開発者の都合ではなく、実際の使用者の体験を重視

## 📊 開発Phase概要

### Phase 1: 基本表示機能（1週間）
**目標**: chokidarを使用したリアルタイムファイル監視の基本実装
- ファイル追加・変更・削除の検出
- コンソール出力
- 基本的な設定構造

### Phase 2: Scan機能（1週間）
**目標**: 起動時の既存ファイルスキャン機能追加
- 既存ファイルの検出
- データベース初期化
- プログレス表示

### Phase 3: Move/Rename検出（2週間）
**目標**: ファイルの移動・リネームを正確に検出
- inode/内容ハッシュによる追跡
- unlink→addペアの検出
- 誤検出防止メカニズム

### Phase 4: Unique表示 & 詳細画面（1週間）
**目標**: 重複イベントの集約とファイル詳細情報表示
- 同一ファイルのイベント集約
- ファイル詳細情報パネル
- イベント履歴表示

### Phase 5: Filter機能（1週間）
**目標**: イベントタイプとファイル名によるフィルタリング
- イベントタイプマスク
- 正規表現フィルタ
- パスフィルタ

### Phase 6: Stats機能（2週間）
**目標**: ファイル活動の統計情報計算と表示
- イベント統計
- ホットファイルランキング
- 時間帯別分布
- ソート機能

## 🏗️ 技術スタック

### コア技術
- **ファイル監視**: chokidar
- **データベース**: SQLite3（WALモード）
- **CLI**: リアルタイムストリーム表示
- **テスト**: E2E重視（RDD方式）

### Cache戦略（段階的強化）
- **Phase 1-2**: 基本メモリキャッシュ
- **Phase 3-4**: LRUキャッシュ導入
- **Phase 5-6**: 永続化キャッシュ（SQLite統合）

### パフォーマンス目標
- 起動時間: < 1秒（1万ファイル環境）
- メモリ使用量: < 200MB（通常使用時）
- CPU使用率: < 5%（アイドル時）
- イベント検出遅延: < 100ms

## 📋 リスク管理

### 技術的リスク
1. **chokidarの制限事項**
   - プラットフォーム依存の挙動
   - 大量ファイルでのパフォーマンス
   - moveイベントの非サポート

2. **スケーラビリティ**
   - 10万ファイル以上での動作
   - リアルタイム性の維持
   - メモリ使用量の増大

### 対策
- 早期プロトタイプでの検証
- 代替ライブラリの調査（node-watch等）
- 段階的な最適化

## 🎯 成功指標

### 定量的指標
- **動作可能日数**: 連続して`npm start`が成功した日数
- **ロールバック率**: 動作しないためロールバックした回数
- **実ユーザーフィードバック**: 実際に使用した人からの評価

### 定性的指標
- 直感的なUI
- 安定した動作
- 有用な統計情報
- 拡張しやすい設計
