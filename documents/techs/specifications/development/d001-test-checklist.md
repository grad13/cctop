# テスト実装チェックリスト (t000)

**作成日**: 2025-06-21  
**作成者**: Inspector Agent  
**目的**: 機能実装時に必ず確認するテスト作成チェックリスト

## 📋 機能実装時の必須チェックリスト

### 1️⃣ テスト作成前の確認
- [ ] 実装する機能の仕様書を読んだか？
- [ ] 仕様書の内容は完全に理解できたか？
- [ ] **仕様が不明瞭な場合、ユーザーに質問したか？**
- [ ] **仕様書を確認してその通りに実装できているか確認したか？**
- [ ] **もし不明な点があればユーザーに確認したか？**
- [ ] テスト対象の境界（スコープ）は明確か？
- [ ] 依存関係（外部API、DB、ファイルシステム）を把握したか？
- [ ] **ディレクトリ構造は../system/a001-directory-structure.mdに従っているか？**

### 2️⃣ テストケース設計
#### 正常系
- [ ] 典型的な使用例をカバーしているか？
- [ ] 期待される出力が明確に定義されているか？
- [ ] 複数の正常パターンを網羅しているか？

#### 異常系
- [ ] null/undefined の入力を処理できるか？
- [ ] 空配列/空オブジェクトを処理できるか？
- [ ] 不正な型の入力でエラーを返すか？
- [ ] 例外が適切にthrowされるか？

#### 境界値
- [ ] 最小値（0、空文字、最小配列）でテストしたか？
- [ ] 最大値（上限値、最大文字列長）でテストしたか？
- [ ] 境界値の±1でテストしたか？

### 3️⃣ テスト実装
#### 構造
- [ ] describe/it で適切に構造化されているか？
- [ ] テスト名は何をテストしているか明確か？
- [ ] Arrange-Act-Assert パターンに従っているか？

#### モック・スタブ
- [ ] 外部依存はモック化されているか？
- [ ] モックは必要最小限か？（過度なモックは避ける）
- [ ] モックの振る舞いは現実的か？

#### クリーンアップ
- [ ] beforeEach/afterEach で適切にセットアップ/クリーンアップしているか？
- [ ] テスト間で状態が共有されていないか？
- [ ] リソース（DB接続、ファイル等）は解放されているか？

### 4️⃣ 特定機能のテスト

#### データベース関連
- [ ] トランザクションのロールバックをテストしたか？
- [ ] 同時実行時の競合状態をテストしたか？
- [ ] インデックスが効いているか確認したか？

#### ファイル操作関連
- [ ] 存在しないファイルへのアクセスをテストしたか？
- [ ] 権限がない場合の動作をテストしたか？
- [ ] 大きなファイルでのパフォーマンスを確認したか？

#### CLI/UI関連
- [ ] キーボード入力のシミュレーションは正確か？
- [ ] 画面表示の検証は適切か？
- [ ] エスケープシーケンスは考慮されているか？

### 5️⃣ パフォーマンステスト
- [ ] 大量データ（1000件以上）で動作確認したか？
- [ ] メモリリークがないか確認したか？
- [ ] 実行時間は許容範囲内か？

### 6️⃣ テスト実行・確認
- [ ] すべてのテストが通ったか？
- [ ] カバレッジは80%以上か？
- [ ] CI環境でも動作するか？
- [ ] 他のテストに影響を与えていないか？

### 7️⃣ ドキュメント更新
- [ ] テストで判明した仕様を仕様書に反映したか？
- [ ] 制限事項や注意点を記載したか？
- [ ] 使用例（example）を追加したか？
- [ ] **cctop/README.mdを更新したか？**（新機能・変更点の反映）

### 8️⃣ UI整合性確認（重要）
- [ ] **UIに表示されるキーバインドがすべて実装されているか？**
- [ ] **表示される機能説明と実際の動作が一致するか？**
- [ ] **キーボードショートカットの統合テストを作成したか？**
- [ ] **ヘルプテキストと実装の整合性を確認したか？**
- [ ] **エラーメッセージが実際の処理と一致するか？**

## 🚨 よくある見落とし

### データ関連
- [ ] タイムゾーンの考慮
- [ ] 文字エンコーディング（UTF-8以外）
- [ ] 改行コードの違い（LF/CRLF）

### 非同期処理
- [ ] Promise の reject ケース
- [ ] タイムアウト処理
- [ ] 競合状態（race condition）

### エッジケース
- [ ] 循環参照
- [ ] 再帰の深さ制限
- [ ] メモリ不足時の動作

## 📁 ディレクトリ構造の確認

### 実装時の必須確認
- [ ] **ファイルは正しいディレクトリに配置されているか？**
  - cli/ - CLI表示関連
  - analyzers/ - 分析・統計処理
  - monitors/ - ファイル監視
  - database/ - データベース関連
  - ui/ - 表示コンポーネント（低レベル）
- [ ] **テストファイルは対応するディレクトリ構造に配置されているか？**
  - test/unit/cli/ - CLIのテスト
  - test/unit/analyzers/ - アナライザーのテスト
  - test/unit/monitors/ - モニターのテスト
  - test/unit/database/ - データベースのテスト
- [ ] **不要なファイル・ディレクトリが残っていないか？**
  - tmp/などの一時ディレクトリ
  - 古いファイル名の残骸

## 📝 テストレビューチェックリスト

### コード品質
- [ ] テストコードは読みやすいか？
- [ ] 重複したテストはないか？
- [ ] テストの意図は明確か？

### 保守性
- [ ] テストデータは外部化されているか？
- [ ] マジックナンバーは定数化されているか？
- [ ] 将来の変更に対して脆弱でないか？

---

**使い方**: 
1. 機能実装開始時にこのチェックリストをコピー
2. 該当する項目をチェックしながら進める
3. すべてチェックが付いたらレビュー依頼

**注記**: このチェックリストは最低限の品質を保証するためのものです。機能に応じて追加の確認項目を設けてください。
