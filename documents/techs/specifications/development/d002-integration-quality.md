# 統合品質チェックリスト (t001)

**作成日**: 2025-06-22  
**作成者**: Inspector Agent  
**目的**: t000の致命的欠陥を修正した実用性重視の品質保証  
**問題**: t000は単体テスト中心で実際の動作確認が欠如

## 🚨 t001の絶対原則

### **1. 実装完了の新定義**
実装完了 = **エンドユーザーが実際に使用できる状態**

### **2. 品質評価の順序（必須）**
1. **実環境動作確認** ← 最優先
2. 統合テスト確認
3. 単体テスト確認
4. ドキュメント整合性

### **3. 失敗時の即座停止**
実環境動作で1つでも失敗 → 全評価停止・修正まで進行禁止

## 🚨 t001失敗時の対応

### 失敗レベル定義
- **Level 1**: 実環境動作確認で1項目でも失敗 → **実装品質不適格**
- **Level 2**: 統合テストで3項目以上失敗 → **設計見直し必要**
- **Level 3**: パフォーマンス目標未達 → **最適化必要**

### 失敗時の必須対応
1. **即座に作業停止** - 追加実装・評価の禁止
2. **根本原因分析** - 表面的修正の禁止
3. **修正完了まで「未完成」扱い** - 品質評価レポート作成禁止
4. **修正後のt001再実行** - 全項目の再確認必須

## 🔄 t000との違い

### t000の問題点
1. 単体テスト中心で実環境確認なし
2. 統合レベルの検証が不十分
3. 実用性より理論的完璧性重視
4. モックによる実装バグの隠蔽（r018発見）

### t001の改善点
1. **実環境動作確認を最優先**
2. **実際の統合ポイントを重点確認**
3. **エンドユーザー視点での品質評価**
4. **失敗時の即座停止機構**
5. **モックと実装の乖離検出（r018教訓）**

## ⚠️ t001使用時の必須注意事項

### 評価者への警告
1. **実環境確認を絶対に怠るな** - チェックリスト項目の形式的確認は無意味
2. **1項目でも失敗したら停止** - 「だいたい動く」は不合格
3. **体感的な改善を確認せよ** - 数値だけでなく実際の使用感も評価
4. **長期安定性を軽視するな** - 短時間の動作確認だけでは不十分

### 開発者への警告
1. **実装 ≠ 完成** - t001合格までは未完成品
2. **単体テスト合格は前提条件** - t001の前にt000レベルは必須
3. **統合設計を最初から考慮** - 後付け統合は高確率で失敗
4. **パフォーマンス目標を明確化** - 体感的改善を数値で定義
5. **モックは実装の完全コピー** - 便利メソッドの追加は厳禁（r018教訓）

## 🚫 t001で絶対禁止の行為

### 評価者への厳重警告
1. **t001で始まるファイル名作成禁止** - t001_evaluation_config.md、test_t001_complete.js等は混乱の元
2. **DIRECTORY-STRUCTURE違反ファイル作成禁止** - cctop/直下のtest_*.js、レポート配置等は仕様書違反
3. **確認なしでの勝手な行動禁止** - 不明点への質問なしで追記・実装等を独断実行厳禁
