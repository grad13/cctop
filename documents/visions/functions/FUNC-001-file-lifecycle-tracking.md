# FUNC-001: ファイルライフサイクル追跡機能

**作成日**: 2025年6月25日 16:00  
**更新日**: 2025年6月25日 19:00  
**作成者**: Architect Agent  
**ステータス**: Active  
**Version**: 0.2.0.0

## 📊 機能概要

ファイルの全ライフサイクルを6つのイベントタイプ（find/create/modify/move/delete/restore）で追跡する包括的システム。chokidarイベントとの正確なマッピングにより、削除・復活・移動を含むあらゆるファイル操作を記録。

**ユーザー価値**: 
- 完全なファイル変更履歴の可視化
- 削除ファイルの復活検出（restore/refind）
- システム再起動後の状態整合性確認
- 監視範囲境界での正確なイベント判定

## 🎯 機能境界

### ✅ **実行する**
- **ファイルのみ対象**（ディレクトリは除外）
- delete/restoreイベントの定義・記録（lost/refind廃止）
- inode再利用対応のオブジェクト同一性管理
- ファイル復活・復元の自動検出
- システム再起動後の状態検証・同期
- 復活検出期間の設定（restore: デフォルト5分、設定可）

### ❌ **実行しない**
- **ディレクトリの追跡**（複雑性に対して価値が低い）
- **ファイルのコピー検出**（copyイベントは追跡対象外）
- **削除理由の識別**（実際の削除と監視範囲外への移動は区別しない）
- ファイル内容の完全バックアップ
- 削除ファイルの自動復元機能
- 複雑なバージョン管理システム

## 📋 イベントタイプ定義

### **イベントタイプ一覧**

| イベント名 | 説明 |
|-----------|------|
| **find** | 初期スキャン時の既存ファイル発見 |
| **create** | リアルタイム監視中の新規ファイル作成 |
| **modify** | ファイル内容・メタデータの変更 |
| **move** | ファイルの移動・リネーム |
| **delete** | ファイル不在の検出（削除・監視外移動・起動時不在） |
| **restore** | delete状態からの復活検出 |

### **イベント統合の設計判断**

**lost/refindイベントの廃止理由**：
- 「ファイルが存在しない」という事実を異なる状態で表現する複雑性を排除
- scan時の不在もリアルタイム監視時の削除も、本質的には同じ「不在」
- 統一により実装とユーザー理解の両方がシンプル化

### **イベントで活用可能な情報**

| イベント | inode取得 | stats取得 | DB参照内容 | パス取得元 | タイムスタンプ | 特記事項 |
|---------|----------|----------|-----------|-----------|--------------|---------|
| **find** | ✅ 可能 | ✅ 可能 | ❌ 不要 | ✅ 現在のFS | ✅ 正確 | 一括処理される |
| **create** | ✅ 可能 | ✅ 可能 | deleted状態 | ✅ 現在のFS | ✅ 正確 | restore判定必要 |
| **modify** | ✅ 可能 | ✅ 可能 | active状態確認 | ✅ 現在のFS | ✅ 正確 | 高頻度発生 |
| **move** | ⚠️ 新:可/旧:DB | ⚠️ 新:可/旧:不可 | 旧パスのinode | ✅ 旧:DB/新:FS | ✅ 正確 | unlink→add検出 |
| **delete** | ❌ 不可 | ❌ 不可 | active object_id | ✅ 現在のFS/DB | ✅ 正確 | unlinkまたは起動時不在 |
| **restore** | ✅ 可能 | ✅ 可能 | deleted状態確認 | ✅ 現在のFS | ✅ 正確 | 時間制限あり |

**凡例**:
- ✅ 可能/必須: 確実に取得・利用可能
- ⚠️ 条件付き: 特定の条件下でのみ可能
- ❌ 不可/不要: 取得不可能または不要

### 📋 chokidarイベントマッピング定義

各イベントタイプがchokidarのどのイベントにマッピングされるかの正確な定義。

| cctopイベント | chokidarイベント | 監視フェーズ | タイミング条件 |
|--------------|----------------|------------|--------------|
| **find** | `add` | 初期スキャン | ready前 + `ignoreInitial: false` |
| **create** | `add` | リアルタイム監視 | ready後 |
| **modify** | `change` | リアルタイム監視 | ready後 |
| **move** | `unlink` → `add` | リアルタイム監視 | 100ms以内 + 同一inode |
| **delete** | `unlink` | リアルタイム監視 | moveタイムアウト後 |
| **delete** | なし | 初期スキャン | DBに存在 but FSに不在 |
| **restore** | `add` | リアルタイム監視 | ready後 + 削除履歴5分以内（設定可） |

**注**: ディレクトリ関連のイベント（`addDir`/`unlinkDir`）は対象外。ファイルのみを追跡対象とする。

主な状態：

- **Active**: 監視対象として活動中（最も一般的）

## 📊 ファイル追跡の実装課題と解決方針

### 1. **同一性管理の課題**

#### delete → restore時のinode不一致
- **課題**: 削除後に同パスで再作成されたファイルは新しいinodeを持つ
- **解決方針**: 
  - パス + サイズ + 時間窓（5分）での同一性判定
  - ファイルサイズ（バイト単位）をメタデータとして常に記録
  - object_idは維持し、新inodeは新規エントリとして記録
  - 表示時は同一ファイルの継続として扱う

#### move検出の不確実性
- **課題**: unlink→addの順序保証なし、タイミングも不定
- **解決方針**:
  - pending_unlinksマップで一時的に保持（表示は遅延）
  - inode一致 + 時間閾値（デフォルト100ms・設定可）で判定
  - タイムアウト後はdeleteとして確定・表示
  - inode取得不可の場合はdelete & createとして処理

- **制限事項**:
  - ファイル名の入れ替え操作は、タイミングによっては正しく検出できない可能性あり

### 2. **履歴管理の設計**

#### オブジェクト履歴の保持
```
files (永続的なファイル識別)
├── id (file_id)
├── inode
└── is_active (boolean)

events (全イベント記録)
├── file_id (FK)
├── event_type
├── file_path
├── inode (その時点の値)
└── timestamp
```

**is_activeのboolean管理の利点**:
- delete検出: unlinkイベント。もしくは起動時に`is_active = true`のファイルの存在をチェックし、しないものをdeleteとする
- restore判定: リアルタイム監視で`is_active = false`の中から一致するinodeがないかチェック
- シンプルなインデックスで高速な状態フィルタリングが可能

#### 状態遷移の記録方針
- **全イベントを記録**: 削除も含めて完全な履歴を保持
- **状態は更新**: filesテーブルのis_activeフィールドで現在状態を管理
- **復活判定**: 最新のdeleteイベントからの経過時間で判定

### 3. **表示の一貫性**

#### 詳細モードでの表示ルール
- **file_id単位で集約**: 同一ファイルの全ライフサイクルを1つの履歴として表示
- **inode変化の明示**: restore/moveでinodeが変わった場合は視覚的に区別
- **時系列の保証**: タイムスタンプ順で確実にソート

### 4. **監視範囲境界での一貫性**

| シナリオ | 検出 | 表示方針 |
|---------|------|----------|
| 監視外へ移動 → 監視内へ復帰 | delete → create/restore | 時間窓内なら同一ファイルとして表示 |
| 一時的な監視外経由の移動 | delete → create | moveとして推測表示（オプション） |

## 🧪 テスト要件

### **基本イベント追跡テスト**
- [ ] find: 初期スキャンでの既存ファイル検出
- [ ] create: リアルタイム新規ファイル作成
- [ ] modify: ファイル内容変更の正確な検出
- [ ] delete→同パス即座create→restore検出
- [ ] システム再起動→delete検出→restore検出

### **高度追跡テスト**
- [ ] move: 連続unlink→addでの移動検出
- [ ] inode再利用での適切な新規オブジェクト作成
- [ ] 複雑な移動→削除→復活シナリオ

### **復活検出テスト**
- [ ] restore: delete状態からの復元（時間制限）
- [ ] 起動時delete→復活検出
- [ ] 大量ファイル削除→部分復活の追跡

### **パフォーマンステスト**
- [ ] 10,000ファイル削除→delete検出性能
- [ ] 同一性判定の応答時間（<10ms）
- [ ] メモリ使用量の適正性
