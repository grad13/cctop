## 📊 管理方針

### パイロット機能の位置づけ
- **実験段階**: アイデア検証・仕様策定中
- **変更自由**: 大幅な仕様変更・削除も可能
- **実装対象外**: Builder/Validatorの作業対象外
- **将来性重視**: 革新的・実験的アプローチを推奨

### 昇格基準
1. **仕様安定性**: 基本仕様が固まり大きな変更がない
2. **実装可能性**: 技術的実現性が確認済み
3. **ユーザー価値**: 明確な価値提案と使用シナリオ
4. **設計整合性**: 既存システムとの整合性確保

### 昇格プロセス
1. **Draft完成**: 仕様書の完全性確認
2. **技術検証**: 実装可能性の確認
3. **Architect承認**: 設計品質・整合性の確認
4. **functions/移行**: Active機能として正式採用
5. **番号再割当**: 適切なカテゴリ番号への変更

## 🎯 品質基準

### パイロット機能の要件
- **創造性**: 既存機能を超える革新的アプローチ
- **実用性**: 実際の使用場面での価値提供
- **技術的先進性**: 最新技術・手法の活用
- **拡張性**: 将来の機能拡張への対応

### 記録すべき内容
- **実験意図**: なぜこの機能を試すのか
- **仮説検証**: どのような仮説を検証するか
- **代替案検討**: 他のアプローチとの比較
- **リスク評価**: 実装・運用時のリスク

## 🔗 他ディレクトリとの関係

### functions/との関係
- **成果移行**: 成熟したパイロット機能の昇格先
- **技術参照**: 既存Active機能との整合性確保
- **設計継承**: 確立された設計パターンの活用

### blueprints/との関係
- **実装計画**: パイロット機能の実装ロードマップ
- **統合設計**: 複数パイロット機能の組み合わせ設計

### supplementary/との関係
- **実装指針**: パイロット機能の実装方針
- **技術検証**: 実装アプローチの検証結果

## ⚠️ 注意事項

### 開発制限
- **実装禁止**: Builder/Validatorは作業対象外
- **設定分離**: config.jsonへの混入禁止
- **依存関係注意**: Active機能への影響回避

### 品質保証
- **定期見直し**: 月次でのパイロット機能評価
- **不要機能削除**: 価値のない機能の積極的削除
- **重複防止**: 類似機能の統合・整理
