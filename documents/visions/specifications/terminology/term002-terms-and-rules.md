**作成日**: 2025年6月14日

**目的**: プロジェクト全体で使用する用語の統一と明確化  
**参照URL**: documents/visions/specifications/terminology/（用語定義）

## 📝 基本用語定義

### 問題分類の用語

#### インシデント (Incident)
**定義**: メタ手続きに関する問題・違反  
**対象範囲**:
- プロセス違反（チェックリスト未実行、手順スキップ等）
- ルール違反（CLAUDE.md、仮説、プロトコルの未遵守）
- ドキュメント管理問題（更新忘れ、構造違反等）
- 仮説管理問題（記録漏れ、評価不備等）
- エージェント権限違反

**対応方法**: インシデント対応チェックリスト（5フェーズ）による体系的対応

**記録場所**: `documents/records/incidents/`

**例**:
- status/{agent}.md更新忘れ
- 仮説一覧表への記載漏れ
- インシデント対応手順の不完全実行
- エージェント権限外アクセス

#### バグ (Bug)
**定義**: コードに関する問題  
**対象範囲**:
- 機能不動作（期待動作と実際の動作の相違）
- パフォーマンス問題
- UI/UXの不具合
- セキュリティ脆弱性
- データ不整合

**対応方法**: H013原則に基づく根本原因分析と修正

**記録場所**: 
- 緊急度高: `documents/records/incidents/`（バグだがインシデント扱い）
- 通常: GitHub Issues またはdocuments/agents/status/{agent}.md

**例**:
- TaskGrid入力不可
- Quick Switchフォーカス問題
- タイマー表示バグ
- データ同期エラー

## 🔄 分類の判断基準

### インシデントとして扱う場合
1. **プロセス・ルール違反**がある
2. **メタレベル**の問題（仕組み・手順の問題）
3. **複数回発生**または**体系的問題**の可能性
4. **プロジェクト運営**に直接影響

### バグとして扱う場合
1. **技術的実装**の問題
2. **ユーザー機能**に直接影響
3. **コードレベル**での修正が必要
4. **単発的**な技術問題

### 境界ケース
**技術的問題だがインシデント扱いするもの**:
- H013違反（対症療法的修正）
- セキュリティ関連（H014違反）
- プロセス違反を伴う技術問題

### 開発管理の用語

#### ルール (Rule)
**定義**: 厳格に守るべき規則・原則  
**特徴**: 違反時はインシデント扱い  
**例**: 
- CLAUDE.mdの各種ルール
- H025（即時記録プロトコル）
- エージェント権限規定

#### ガイドライン (Guideline)
**定義**: 推奨される手順・方針  
**特徴**: 柔軟性があり、状況に応じて調整可能  
**例**:
- 開発ガイドライン
- コードレビュー基準
- 文書作成の推奨事項

#### プロトコル (Protocol)
**定義**: 確立された手順・プロセス  
**特徴**: 体系的な実行手順を定義  
**例**:
- インシデント対応プロトコル（5フェーズ）
- デプロイメントプロトコル
- 質問解決プロトコル

### エージェント関連の用語

#### Agent (大文字)
**使用法**: 固有名詞として（Coder Agent, Clerk Agent, Inspector Agent）  
**例**: "Coder Agentが実装を担当"

#### agent (小文字)
**使用法**: 一般名詞として  
**例**: "各agentのdocuments/agents/status/{agent}.mdを更新"

## 📊 対応レベル

### インシデント対応
1. **即座に全作業停止**
2. **5フェーズ強制実行**
3. **根本原因分析必須**
4. **再発防止策の実装**

### バグ対応
1. **緊急度に応じた優先度設定**
2. **H013原則適用**（対症療法禁止）
3. **根本修正の実装**
4. **テスト・検証**

## 📈 用語使用の原則

1. **一貫性**: 同じ問題には同じ用語を使用
2. **明確性**: 曖昧な表現を避け、定義に基づく分類
3. **適切性**: 問題の性質に応じた適切な対応レベル
4. **学習**: 用語の使い分けから問題の本質を理解
