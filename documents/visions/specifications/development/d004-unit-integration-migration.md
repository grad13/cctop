# Unit Test to Integration Test Migration Checklist

このチェックリストは、単体テストから統合テストに移行した機能と、それらの統合テストで実際に検証されているかを追跡するためのものです。

## 移行済みの機能

### 1. JavaScript設定ファイルの読み込み (`config-loader.js`)

#### 単体テストで簡略化したもの：
- [ ] JS設定ファイルの実際の読み込み
- [ ] require.cacheのクリア機能
- [ ] 関数型エクスポートの処理
- [ ] 無効なJSファイルのエラーハンドリング

#### 統合テストで検証すべき内容：
- [ ] `test/integration/config-js-loading.test.js`で実際のファイルシステムを使用した読み込み
- [ ] ファイル更新後のキャッシュクリアが正しく動作すること
- [ ] 関数型エクスポート（`module.exports = () => {}`）のサポート
- [ ] 構文エラーがあるJSファイルの適切なエラーメッセージ

### 2. UIレンダラーモジュール (`stream-renderer.js`, `buffered-renderer.js`)

#### 単体テストで問題があったもの：
- [ ] ファイルパスが`src/ui/`から`src/renderers/`に移動
- [ ] モジュールの実際の動作検証

#### 統合テストで検証すべき内容：
- [ ] `test/integration/ui-keyboard-integration.test.js`で実際のレンダリング動作
- [ ] キーボード入力との連携
- [ ] 画面更新の正確性

### 3. バックグラウンドキャッシュ機能

#### 単体テストで簡略化したもの：
- [ ] 非同期バックグラウンドローディングの動作
- [ ] キャッシュウォーミングのタイミング

#### 統合テストで検証すべき内容：
- [ ] `test/integration/background-cache-integration.test.js`で実際のDB操作を含むキャッシュ動作
- [ ] 初回リクエスト時の即座レスポンス
- [ ] バックグラウンドでの追加データロード
- [ ] 複数イベントタイプのフィルタリング

### 4. 統計キャッシュ機能

#### 単体テストで簡略化したもの：
- [ ] TTLベースのキャッシュ有効期限
- [ ] キャッシュヒット/ミスの正確なカウント

#### 統合テストで検証すべき内容：
- [ ] `test/integration/statistics-cache-integration.test.js`で実際のDB統計クエリのキャッシュ
- [ ] getTopActiveObjects、getEventTypeCounts、getDailyStatisticsのキャッシュ動作
- [ ] キャッシュメトリクスの正確性

### 5. 境界値とパフォーマンステスト

#### 単体テストで検証が難しいもの：
- [ ] 大量データ（1000件）の挿入パフォーマンス
- [ ] 大きなファイル（10MB）のハッシュ計算
- [ ] 並行処理での競合状態

#### 統合テストで検証すべき内容：
- [ ] 実際のファイルシステムとDBを使用した境界値テスト
- [ ] パフォーマンス測定（時間計測）
- [ ] メモリ使用量の監視

## 統合テストの実装状況

### 完全に実装済み ✅
- [x] JavaScript設定ファイルの基本的な読み込み
- [x] UIモジュールのパス修正

### 部分的に実装済み ⚠️
- [ ] バックグラウンドキャッシュ（初期データのプリロードが必要）
- [ ] 統計キャッシュ（メトリクスの期待値調整が必要）

### 未実装 ❌
- [ ] 境界値テストの一部（実際のファイルシステム操作）
- [ ] パフォーマンステストの時間計測精度

## 推奨アクション

1. **統合テストの事前データ準備**
   - キャッシュテストでは、テスト開始前に少量のデータをプリロードする
   - 統計テストでは、既知のデータセットを用意する

2. **期待値の柔軟性**
   - キャッシュヒット/ミス数は`toBe()`ではなく`toBeGreaterThanOrEqual()`を使用
   - タイミング依存のテストには適切な待機時間を設定

3. **モックの最小化**
   - 統合テストではできるだけ実際のモジュールを使用
   - 外部依存（ファイルシステム、DB）のみモック化を検討

4. **エラーケースの網羅**
   - ファイルが存在しない場合
   - DBが利用できない場合
   - 権限エラーなど

## 次のステップ

1. このチェックリストに基づいて、統合テストの不足部分を特定
2. 優先度の高い機能から統合テストを追加
3. 単体テストは「ユニットの振る舞い」に焦点を当て、統合テストは「実際の動作」を検証する役割分担を明確化
