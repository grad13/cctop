# 問題分析・改善提案チェックリスト

**作成日時**: 2025年6月14日 09:15  
**最終更新**: 2025年6月19日（REP-0080対応）  
**目的**: 問題分析・改善提案時の体系的アプローチを確実に実行する  

## 📝 問題分析・改善提案必須手順

### Phase 1: 問題分析（10分以内）
- [ ] **問題の明確化**
  - [ ] 発生した問題を1-2文で記述
  - [ ] 影響範囲を特定
  - [ ] 緊急度を判定

- [ ] **原因分析**
  - [ ] 5 Whys法で根本原因を探る
  - [ ] 複数の原因候補を列挙
  - [ ] 最も可能性の高い原因を特定

### Phase 2: 改善提案立案（15分以内）
- [ ] **改善提案の作成**
  - [ ] 「現状の問題：〜」「改善案：〜」「期待効果：〜」形式で記述
  - [ ] 実装可能で検証可能な内容であることを確認
  
- [ ] **検証計画**
  - [ ] 成功指標を定義（定量的に）
  - [ ] 検証期間を設定（通常1週間）
  - [ ] 測定方法を明確化

### Phase 3: 文書化・記録（10分以内）
- [ ] **レポート作成**
  - [ ] records/reports/にREP-XXXX形式で作成
  - [ ] 必須項目をすべて記載
    - [ ] 作成日時・作成者
    - [ ] 問題の背景・現状
    - [ ] 分析結果・根本原因
    - [ ] 改善提案・実装計画
    - [ ] 期待される効果
    - [ ] 検証方法

### Phase 4: records/reports/README.md更新（5分以内）★重要★
- [ ] **一覧表への追加**
  - [ ] 新規レポートをリストに追加
  - [ ] REP番号、タイトル、作成日、ステータス、ファイル名を記載
  
- [ ] **メタ情報更新**
  - [ ] 最終更新日を今日の日付に変更
  - [ ] 合計レポート数を更新

### Phase 5: 関連更新（5分以内）
- [ ] **status/{agent}.mdへの記録**
  - [ ] 分析・改善提案作成を作業ログに記載
  - [ ] 問題と対策の概要を記録
  
- [ ] **CLAUDE.md確認**
  - [ ] 新しいルールが必要な場合は追加
  - [ ] 既存ルールとの整合性確認

## ⚠️ よくある失敗と対策

### 失敗パターン
1. **README.md更新忘れ**
   - 仮説ファイルだけ作って満足してしまう
   - 対策：このチェックリストのPhase 4を必ず実行

2. **番号の重複**
   - README.mdを確認せずに番号を付ける
   - 対策：必ず最新番号を確認してから採番

3. **検証期間の未設定**
   - 「永続」や期限なしにしてしまう
   - 対策：原則1週間、最長でも1ヶ月に設定

## 🎯 完了確認

すべてのチェックボックスにチェックが入っていることを確認してから作業完了とする。特に**Phase 4のREADME.md更新は必須**。
