**目的**: インシデント発生時の体系的対応を確実に実行する  

## ⚠️ 必須参照宣言

**このチェックリストは必ず毎回参照すること**
- 記憶に頼った対応は厳禁
- 参照なしでの対応は二次インシデント扱い（INC-20250614-014）
- 「知っている」は参照省略の理由にならない

## 🔐 権限・責任分担

### Clerk Agent権限
- **CLAUDE.md編集権限**: 全面的な編集・ルール追加権限
- **最終的な責任**: インシデント対応のプロセス改善・ルール策定

### 他エージェント権限（権限制限あり）
- **CLAUDE.md**: 読み取り専用・編集権限なし
- **対応範囲**: protocols/、チェックリスト等の作業エリア
- **Clerk報告**: CLAUDE.md更新が必要な場合はClerkに報告

## 🚨 インシデント対応必須手順

### Phase 1: 初期記録（5分以内）
- [ ] **インシデント記録作成**
  - [ ] 発生日時・報告者・対応者を記載
  - [ ] インシデントIDを採番（INC-YYYYMMDD-XXX）
  - [ ] 概要を1-2文で記載
  - [ ] status/{agent}.mdに記録

### Phase 2: 詳細分析（15分以内）
- [ ] **現象の詳細記述**
  - [ ] 何が起きたか（事実のみ）
  - [ ] 期待動作と実際の差異
  - [ ] 影響範囲・重要度
  
- [ ] **関連文書の網羅的確認**（INC-20250614-021対策）
  - [ ] 関連キーワードでの全文書検索実施
  - [ ] CLAUDE.md内での関連記述確認
  - [ ] 文書整合性の検証
  
- [ ] **原因分析（5 Whys法）**
  - [ ] 直接原因：なぜ起きたか？
  - [ ] 第2層：その原因は何か？
  - [ ] 第3層：さらにその原因は？
  - [ ] 第4層：もっと深い原因は？
  - [ ] 根本原因：最も本質的な原因

### Phase 3: 対策立案（20分以内）
- [ ] **即時対応**
  - [ ] 現在の問題を解決する行動
  - [ ] 影響を最小化する措置
  
- [ ] **再発防止策**
  - [ ] プロセス改善案
  - [ ] ルール・仕組みの追加
  - [ ] 仮説立案（必要に応じて）

### Phase 4: 実装・記録（30分以内）
- [ ] **対策の実装**
  - [ ] CLAUDE.md更新（ルール追加）※Clerk Agent権限
  - [ ] 仮説ファイル作成（H番号）
  - [ ] チェックリスト更新
  - [ ] meta/protocols/ディレクトリ更新（エージェント作業時）
  
- [ ] **記録の完了**
  - [ ] incidents/README.md更新
  - [ ] status/{agent}.md更新
  - [ ] 関連文書のリンク追加

### Phase 5: 検証（翌日以降）
- [ ] **効果測定**
  - [ ] 同種インシデントの発生有無
  - [ ] 対策の有効性評価
  - [ ] 追加対策の必要性判断

## ⚠️ 重要な原則

### やってはいけないこと
- ❌ 記録だけして終わる
- ❌ 表面的な原因分析で満足する
- ❌ 対症療法だけで根本解決しない
- ❌ 対策を立案だけして実装しない

### 必ずやること
- ✅ 5 Whysで根本原因まで掘り下げる
- ✅ 再発防止の仕組みを必ず作る
- ✅ 対策は即座に実装する
- ✅ 効果を後日検証する

## 📊 インシデント分析の例

### 悪い例
- 現象：記録を忘れた
- 原因：忙しかったから
- 対策：次は気をつける

### 良い例
- 現象：status/{agent}.md記録漏れ
- Why1：作業に集中して記録を忘れた
- Why2：記録より作業を優先する習慣
- Why3：記録の重要性が内面化されていない
- Why4：記録を強制する仕組みがない
- 根本原因：プロセスが習慣化を前提としている
- 対策：並行記録プロトコル導入（H025）
