## 🔬 2つのサブシステム

### 1. Protocols（確立済みプロトコル）
- **目的**: 実証済みプロセス・ルールの標準化
- **内容**: 開発プロトコル、用語集、ガイドライン
- **特徴**: 日常的に参照される実践的ルール
- **例**: P000（システム最上位原則）、P007（文書整合性チェック）等

### 2. Checklists（確認事項）
- **目的**: タスク完了・プロセス遵守の確認
- **内容**: 重要作業の確認リスト
- **特徴**: 品質保証・漏れ防止
- **例**: タスク完了チェックリスト、インシデント対応チェックリスト

## 📊 統一管理の利点

1. **一元的把握**: メタレベル活動の全体像を即座に確認
2. **相互参照**: 仮説→プロトコルの流れを追跡
3. **重複防止**: 類似活動の統合・整理が容易
4. **効率向上**: 適切な手法を素早く選択

## 🎯 ライフサイクル

```
問題発生
  ↓
[直接Protocols作成] プロセス・ルール策定（P番号付与）
  ↓
実装・検証
  ↓
日常的な活用・改善
```

**注**: 従来の仮説検証プロセスは終了し、成熟したプロトコル体系に移行しました。

## 📏 管理指標

### 活動量指標
- **プロトコル数**: アクティブなプロトコルの総数（現在: 35個）
- **適用率**: protocolsの実際の使用頻度
- **更新率**: protocolsの定期見直し・更新頻度

### 品質指標
- **問題解決率**: プロトコル適用による改善成功率
- **一貫性**: プロトコル間の整合性維持度
- **ROI**: プロトコル体系の投資対効果

## 🔗 関連システム

- **dominants/**: 不変の公理・最高位原則
- **CLAUDE.md**: 日常的な開発指針（最重要）
