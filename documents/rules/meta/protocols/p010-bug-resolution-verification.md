## 🎯 バグ解決の5段階ステータス定義（H040由来）

### ステータス定義
1. **発見（Found）**: バグが報告・発見された状態
2. **調査中（Investigating）**: 原因分析中
3. **実装中（Implementing）**: 修正実装作業中
4. **実装完了・確認待ち（Awaiting Verification）**: 実装完了、ユーザー確認待ち
5. **解決済み（Resolved）**: ユーザー確認完了、正常動作確認済み

### 必須確認プロセス
```
実装完了 → ユーザーへ報告 → ユーザー動作確認 → 解決判定
```

### 絶対禁止事項
- ❌ ユーザー確認なしでの「解決済み」判定
- ❌ 実装完了のみでのarchive移動
- ❌ status更新時の「解決完了」記載（確認前）

## 📋 バグ解決判定の必須手順

### Step 1: 文書記録確認
- [ ] バグファイル内のステータス確認
- [ ] 修正実装の記録確認
- [ ] コミット記録の存在確認

### Step 2: 技術的検証
- [ ] **実際の修正内容確認**: コードレベルでの修正確認
- [ ] **テスト結果確認**: 修正が動作することの確認
- [ ] **副作用確認**: 修正による他の問題の有無

### Step 3: ユーザー確認（必須）
- [ ] **ユーザー動作確認**: 実際にユーザーが問題解決を確認
- [ ] **再現テスト**: 元の問題が再現しないことの確認
- [ ] **使用感確認**: 正常な動作を体感できることの確認

### Step 4: 最終判定
- [ ] **3段階すべてクリア**: 文書・技術・ユーザー確認完了
- [ ] **解決宣言**: 「解決済み」ステータス変更
- [ ] **archive準備**: 移動準備完了

## ❌ 解決判定NG条件

以下のいずれかに該当する場合は未解決扱い：

### 文書記録の不備
- バグファイルに「解決済み」の明記がない
- 修正内容が具体的に記載されていない
- テスト結果の記録がない

### 技術的未完了
- 修正コードがコミットされていない
- 修正が部分的で根本解決していない
- 関連する副作用が未対応

### ユーザー未確認
- **ユーザーが解決を確認していない**（最重要）
- 再現テストが実施されていない
- 動作確認が曖昧・不十分

## 🚨 Critical判定基準

### ユーザー確認の絶対性
**「ユーザー確認なしの解決判定は無効」**

- 文書上の記録は参考情報
- 技術的修正は必要条件
- **ユーザー確認が充分条件**

### 判定プロセスの厳格化
1. **疑義がある場合は未解決扱い**
2. **確認不十分な場合は追加検証**
3. **ユーザー指摘は最優先で反映**
