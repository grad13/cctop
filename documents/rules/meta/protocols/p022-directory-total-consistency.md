## 1. 概要

ディレクトリ構造の変更時に、README.md記載とファイルシステムの整合性、および文書間の参照整合性を総合的に維持する統合プロトコル。P021の機能を完全に包含し、H030のREADME.md管理要素も統合。

### 主要用語定義
- **broken links**: 存在しないファイルやディレクトリへの参照（最重要修正対象）
- **Type A不整合**: README.mdに記載があるが実在しないファイル/ディレクトリ
- **Type B不整合**: 実在するがREADME.mdに記載がないファイル/ディレクトリ
- **Type C不整合**: パスや名前の誤記・タイポ

## 2. 適用タイミング

以下のいずれかの場合に適用：
1. **ディレクトリ再編後**（移動・削除・新規作成）
2. **月次メンテナンス**
3. **インシデント発生時**（不整合関連）
4. **ユーザー指示時**

## 3. 実施フロー

### Phase 0: Dominants参照整合性チェック（15分）【最高優先度】

**背景**: Dominant原則はプロジェクトの最高位規則であり、参照整合性の破綻はシステム全体の基盤を不安定化させる。P040不変要素保護プロトコルに準拠し、最小限の修正で最大の整合性を確保する。

#### Step 0.1: Dominants参照抽出（5分）
```bash
echo "=== DOMINANTS REFERENCE INTEGRITY CHECK ==="
echo "Extracting all references from dominants/..."

# dominants内のプロトコル・ファイル参照を抽出
echo "Protocol references:"
grep -o "[PH][0-9][0-9][0-9]" documents/rules/dominants/*.md | sort | uniq

echo "File references:"
grep -o "documents/[^)]*\.md" documents/rules/dominants/*.md
```

#### Step 0.2: 参照存在確認（5分）
各参照について：
1. **プロトコル番号**: protocols/README.mdで状況確認（統合・廃止状況）
2. **ファイル参照**: 実際の存在確認（archive検索含む）
3. **状況記載**: 統合・廃止状況が適切に記載されているか

#### Step 0.3: Critical問題の即座修正判定（5分）
- **存在しない参照**: 修正方針決定（削除・置換・一般化）
- **不正確な参照**: 正確な記載への修正
- **P040準拠**: 最小限変更での整合性確保
- **即座修正**: Critical問題は即座に対応

### Phase 1: 事前準備（5分）
```
1. 作業記録開始（status/{agent}.md）
2. 変更内容の確認（何が変更されたか）
3. チェックリスト参照（必要に応じて）
```

### Phase 2: README.md整合性チェック（30分）
#### Step 2.1: 対象README.mdのリストアップ（5分）
```bash
find documents -name "README.md" -type f | sort
```

#### Step 2.2: 各README.mdの構造記載確認（10分）
各README.mdを開き、以下を確認：
- ディレクトリ構成図の有無
- ファイルリストの有無
- サブディレクトリの説明

#### Step 2.3: 実ディレクトリとの照合（15分）
```bash
# 例: documents/README.mdの場合
ls -la documents/
# README.mdの記載と比較
```

#### Step 2.4: 不整合の検出
- **Type A**: README.mdに記載があるが実在しない
- **Type B**: 実在するがREADME.mdに記載がない
- **Type C**: パスや名前の誤記

### Phase 3: P007実施（60-90分）【Dominants強化版】
```
P007: 文書整合性チェック（Dominants優先版）
- Phase 0結果の活用（Dominants問題は既に処理済み）
- 一般プロトコル参照整合性
- broken links発見（存在しないファイルへの参照）
- 廃止済み参照の確認  
- その他の整合性確認
```

### Phase 4: 統合評価（15分）
```
1. 両プロトコルの結果統合
2. 優先度付け（Critical → Important → Info）
3. 修正計画立案
```

### Phase 5: 修正実施（時間は問題数による）
```
1. Critical問題の即時修正
   - broken linksの解決（参照先修正・ファイル作成）
   - README.md不整合の修正
2. Important問題の計画的修正
3. 修正後の再チェック（必要に応じて）
```

### Phase 6: README.md更新義務（H030由来）
不整合検出時は必ず以下を実行：
1. **即座更新**: ディレクトリ変更と同時にREADME.md更新
2. **メタデータ扱い**: .DS_Storeと同様、常に現状を反映
3. **一体作業**: 「ディレクトリ操作 = README.md更新」として扱う

## 8. documents/README.mdメンテナンス責任（H030由来）

**documents/README.mdは documents管理の司令塔として、常に最新かつ正確である必要があります。**

### 📋 メンテナンス必須タイミング
1. **ディレクトリ構造変更時**: 構成図の即座更新（遅延厳禁）
2. **新機能追加時**: 目的別ナビゲーションへの追加
3. **編集パターン追加時**: 必要資料マップへの新パターン追加
4. **プロセス改善時**: チェックリストの見直し・更新

### 🎯 README.md品質基準
- **網羅性**: 全editing scenarioをカバー
- **正確性**: リンク切れ・パス誤りゼロ
- **最新性**: 構造変更の即日反映
- **効率性**: 必要最小限の確認事項に絞り込み

### 🔄 定期メンテナンス義務

1. **週次チェック**（新機能開発開始時）
   - 重複ファイル・ディレクトリの確認
   - リンク切れの検出・修正
   - 命名規則違反の修正

2. **月次整理**（マイルストーン完了時）
   - 完了した `roadmaps/` 項目を `specifications/` に移動
   - 古い `reports/` ログを `archive/` に移動
   - 不要になったファイルの削除
