# Builder/Validatorバグ記録プロトコル

**作成日**: 2025年6月14日 nodata  
**プロトコルID**: P011  
**策定日**: 2025年6月14日  
**最終更新**: 2025年6月18日（Builder/Validator体制対応）  
**対象**: Builder Agent（実装）・Validator Agent（検証）  
**関連システム**: H037（エージェント適応型記録システム）  
**強制力**: pre-commitフック技術的強制

## 🎯 目的

Builder/Validatorエージェント協調によるバグ対応の体系的記録により、以下を実現：
- バグ知識の蓄積・共有
- 再発防止策の確立
- デバッグ効率の向上
- 品質改善プロセスの確立
- Builder（実装）とValidator（検証）の明確な責任分担

## 📋 適用範囲

### 記録必須対象
**Builder/Validatorが発見・対応するすべてのバグ**:
- 実装中のバグ発見（Builder）
- テスト実行中のバグ発見（Validator）
- ユーザー報告バグの調査・修正
- 既存コードの問題発見
- 表示・動作・パフォーマンス問題
- セキュリティ脆弱性

### 記録対象外
- 仕様変更（バグではない機能修正）
- リファクタリング（機能追加・改善）
- パフォーマンス最適化（問題がない場合）

## 🤝 Builder/Validator責任分担

### Builder Agent責務
- バグ修正の実装
- 根本原因のコード修正
- 単体テストの作成・修正
- バグ記録ファイルの作成・更新（実装部分）

### Validator Agent責務
- バグの検証・再現確認
- 修正効果の確認
- 副作用・回帰テスト
- バグ記録ファイルの更新（検証部分）
- 品質ゲートでの最終判定

## ⏰ 記録必須タイミング（改訂：2025年6月15日）

### 0. バグ対応開始前（必須）
- **specifications確認**: 関連する仕様書を必ず読み返す
- **不明瞭・矛盾点の洗い出し**: 仕様の曖昧さ・コンフリクトを特定
- **ユーザー協力判断**: 質問・協力が必要な項目をリストアップ
- **計画書作成**: バグ対応計画書を作成してから実装開始
- **H013準拠**: 根本原因分析と対症療法回避の確認

### 1. バグ発見時（30分以内）
- **内容**: 現象・再現手順・影響範囲を即座記録
- **ファイル**: `documents/records/bugs/[YYYYMMDD]-[タイトル].md`作成（フラット構造）
- **テンプレート**: `documents/records/bugs/bug-report-template.md`使用
- **担当**: 発見者（Builder/Validator）が初期記録作成
- **注**: active/archiveサブディレクトリは2025年6月16日に廃止

### 2. 調査進展時
- **内容**: 原因分析・仮説・検証結果を記録
- **更新**: 既存バグファイルの「🔍 原因分析」セクション更新
- **担当**: Builder（コード分析）・Validator（動作検証）が協調更新

### 3. 修正実装時
- **内容**: 修正内容・テスト結果・副作用確認を記録
- **更新**: 既存バグファイルの「🔧 修正内容」セクション更新
- **担当**: Builder（実装記録）→ Validator（検証結果追記）

### 4. 完了時
- **内容**: 最終的な解決内容・今後の予防策を記録
- **更新**: ステータスを「解決済み」に変更
- **担当**: Validator（最終確認・品質ゲート判定）

## 📝 記録フォーマット

### ファイル命名規則（2025年6月16日更新）
```
documents/records/bugs/[YYYYMMDD]-[簡潔なタイトル].md
例: documents/records/bugs/20250614-timebox-pause-timer-continues.md

新形式（移行中）:
documents/records/bugs/BUG-YYYYMMDD-XXX-title.md
例: documents/records/bugs/BUG-20250615-001-timebox-timer-abnormal.md
```

### 必須記録項目（改訂：2025年6月15日）
0. **バグ対応計画書**: specifications確認結果、根本原因分析、解決方針、質問事項
1. **基本情報**: 発見日時、重要度、ステータス
2. **現象**: 症状、再現手順、影響範囲
3. **原因分析**: 根本原因、関連コード、仮説検証
4. **修正内容**: 実装修正、変更ファイル、テスト結果
5. **確認事項**: 修正効果、副作用チェック、予防策

### 重要度判定基準
- **高**: システム停止、データ損失、セキュリティ問題
- **中**: 機能不全、使用性に大きな影響
- **低**: 軽微な表示問題、非クリティカルな機能

## 🔒 技術的強制メカニズム

### pre-commitフック拡張
**強制チェック対象キーワード**:
- bug, バグ
- fix, 修正
- error, エラー
- issue, 問題
- crash, クラッシュ
- broken, 動作不良

### 強制実行条件
1. **コミットメッセージ検出**: 上記キーワードが含まれる場合
2. **バグ記録確認**: 過去24時間以内の`documents/records/bugs/`新規記録確認
3. **記録なしブロック**: 該当記録がない場合はコミット拒否

### エラー回避方法
```bash
# 1. バグ記録ファイル作成
cp documents/records/bugs/bug-report-template.md documents/records/bugs/20250614-new-bug.md

# 2. バグ詳細記録
# (エディタでファイル編集)

# 3. コミット再実行
git commit -m "fix: バグ修正"
```

## 🔄 Builder/Validator連携フロー

### バグ対応時のhandoffs活用
1. **Validatorがバグ発見時**:
   - `handoffs/pending/builder/`にバグ修正依頼作成
   - バグ記録ファイル作成・初期情報記載
   
2. **Builderが修正完了時**:
   - `handoffs/pending/validator/`に検証依頼作成
   - バグ記録ファイルに実装内容更新
   
3. **Validatorが検証完了時**:
   - `handoffs/completed/`に完了記録移動
   - バグ記録ファイルに検証結果追記・ステータス更新

## 💬 ユーザーとのコミュニケーション

### 質問・協力依頼のタイミング
1. **仕様確認時**: specificationsに曖昧さ・矛盾がある場合
2. **デバッグ中**: 再現困難・原因不明で協力が必要な場合
3. **修正方針決定時**: 複数の解決案があり優先度確認が必要な場合
4. **テスト時**: ユーザー環境でのテストが必要な場合

### 質問の原則
- **具体的に**: 曖昧な質問を避け、YES/NOで答えられるように
- **選択肢提示**: 複数案がある場合は選択肢として提示
- **根拠説明**: なぜその質問が必要かを説明
- **緊急度明記**: 回答の緊急度を明確に伝える

### 協力依頼の例
```markdown
## 🙋 ユーザーへの質問・協力依頼

### 仕様について（Builder/Validator共通）
Q1: TaskGrid仕様書では「完了タスクは自動削除」とありますが、
    TimeBox仕様書では「完了タスクは履歴保持」となっています。
    どちらを優先すべきでしょうか？

### デバッグ協力（主にValidator担当）
Q2: タイマー停止問題が特定条件でのみ発生するようです。
    以下の操作で再現するか確認いただけますか？
    1. TaskGridで3つ以上タスク作成
    2. TimeBoxで2番目のタスクを開始
    3. 1分後にpauseボタンクリック

### 優先度確認（Builder/Validator協議後）
Q3: 以下の修正案のどちらを優先すべきでしょうか？
    A) 一時的な回避策（1時間で実装可能、完全解決ではない）
    B) 根本的な修正（3時間必要、他機能への影響要確認）
```

## 📊 品質基準

### 記録完成度チェックリスト（改訂：2025年6月15日）
- [ ] **specifications確認**: 関連仕様書を読み返し、期待動作を明記
- [ ] **不明瞭点の明確化**: 仕様の曖昧さ・矛盾を特定し記載
- [ ] **質問・協力依頼**: ユーザーへの質問事項を整理して記載
- [ ] **計画書作成**: バグ対応計画書をファイル冒頭に記載
- [ ] **再現可能**: 他の人が再現手順を実行できる
- [ ] **原因明確**: 技術的原因が具体的に記載
- [ ] **修正詳細**: 何をどう変更したかが明確
- [ ] **テスト完了**: 修正効果と副作用の確認済み
- [ ] **予防策**: 同種問題の再発防止策を提示

### 記録品質の段階評価
- **基本**: 現象・修正内容が記録されている
- **標準**: 原因分析・テスト結果も含む
- **優秀**: 予防策・学習事項まで記載
- **模範**: 将来の開発者への教材レベル

## 🔄 運用ルール

### 継続更新義務
- **調査進展時**: 発見・仮説を随時追記
- **修正実装時**: 変更内容・テスト結果を即座記録
- **完了時**: ステータス更新・予防策記載
- **解決後即座移動**: 解決済みバグは速やかに`documents/archives/bugs/`へ移動（**必須**）

### ファイル管理（2025年6月16日更新）
- **1ファイル1バグ原則**: 1つのバグにつき必ず1つの専用ファイルを作成（**絶対制約**）
- **バグ分離義務**: 複数の問題を発見した場合は各問題ごとに個別ファイル作成
- **関連性の扱い**: 密接に関連するバグでも個別ファイルで管理
- **長期案件**: 大きなバグは段階的更新OK（ただし1ファイル内で完結）
- **解決後管理**: 
  - 解決直後: `documents/archives/bugs/`へ即座移動（**必須**）
  - `documents/records/bugs/`: アクティブ（未解決・調査中）バグのみ保持（フラット構造）
- **bugs/ディレクトリ方針**: records/bugs/は最新のバグのみ、archive/bugs/は解決済み知識蓄積
- **構造変更履歴**: 2025年6月16日にactive/archiveサブディレクトリ廃止、フラット構造へ移行

### 他エージェントとの連携
