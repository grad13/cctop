# P025: レポート作成ガイドライン

**作成日**: 2025年6月16日  
**作成者**: Clerk Agent  
**ステータス**: 確立済み  
**参照URL**: documents/records/reports/README.md（レポート一覧管理）  

## 1. 目的

レポート（records/reports/）作成時の明確な規則を定め、情報の整理と検索性を向上させる。

## 2. 基本原則

### 2.1 1レポート1トピックの原則
- **必須**: 1つのレポートには1つのトピックのみを記載する
- **禁止**: 横並びの複数トピックを1つのレポートに混在させない
- **理由**: 明確な責任範囲、検索性の向上、更新の独立性

### 2.2 依存関係の明示
- **上下関係がある場合**: 各トピックごとに独立したレポートを作成
- **依存関係の記載**: 各レポートに「関連文書」セクションを設け、依存関係を明記
- **相互参照**: 親レポート・子レポートの双方向リンクを維持

## 3. レポート構造

### 3.1 必須セクション
```markdown
# REP-XXXX: [トピック名]

作成日: YYYY年MM月DD日  
作成者: [Agent名]  
ステータス: [計画中/調査中/実装中/完了/保留]  

## 1. 概要
[1つのトピックに焦点を絞った説明]

## 2. 関連文書
- **親レポート**: REP-XXXX（存在する場合）
- **子レポート**: REP-YYYY, REP-ZZZZ（存在する場合）
- **関連仕様**: [specifications/へのリンク]
- **関連プロトコル**: [protocols/へのリンク]
```

### 3.2 トピック分割の判断基準
1. **独立性**: 他のトピックと独立して議論・実装可能か
2. **責任範囲**: 異なるエージェントが担当する内容か
3. **ライフサイクル**: 更新タイミングが異なるか
4. **規模**: 単独で1レポートを構成するに十分な内容か

## 4. 実例

### 4.1 良い例（今回の分割）
- **REP-0024**: MCPサーバー統合調査（技術調査）
- **REP-0025**: externalsディレクトリ最適化（実装計画）
→ 異なるトピックを適切に分離

### 4.2 悪い例（避けるべき）
- 「MCPサーバー統合とexternals最適化提案」
→ 2つの独立したトピックが混在

## 5. 依存関係の記載例

### 親レポートでの記載
```markdown
## 関連文書
- **子レポート**: 
  - REP-0021: Validator環境設計書（技術詳細）
  - REP-0022: エージェント間受け渡しシステム（通信設計）
```

### 子レポートでの記載
```markdown
## 関連文書
- **親レポート**: REP-0020: 5エージェント体制移行計画書
- **前提**: 本レポートはREP-0020の詳細設計として作成
```

## 6. 運用ルール

### 6.1 レポート作成時
1. **トピック確認**: 複数トピックが含まれていないか確認
2. **分割判断**: 必要に応じてトピックごとに分割
3. **依存関係整理**: 関連文書セクションを正確に記載

### 6.2 既存レポートの扱い
- **新規作成時**: この規則を適用
- **既存レポート**: 大規模更新時に規則適合性を確認
- **分割が必要な場合**: 新規レポートとして作成し、旧レポートから参照

## 7. 期待効果

1. **検索性向上**: 1レポート1トピックで明確な検索が可能
2. **保守性向上**: 独立した更新が可能
3. **理解容易性**: 焦点が明確で読みやすい
4. **責任明確化**: 各レポートの担当者が明確

## 8. チェックリスト

レポート作成時の確認事項：
- [ ] 1つのトピックに焦点を絞っているか
- [ ] 複数の横並びトピックが混在していないか
- [ ] 依存関係がある場合、適切に分割されているか
- [ ] 関連文書セクションに依存関係が明記されているか
- [ ] REP番号が重複していないか
- [ ] records/reports/README.mdの一覧を更新したか

## 9. 適用範囲

- **即座に適用**: 本プロトコル制定後のすべての新規レポート
- **既存レポート**: 大幅改訂時にガイドラインへの適合性を確認
- **例外**: 緊急対応の記録等、後日整理が必要な場合
