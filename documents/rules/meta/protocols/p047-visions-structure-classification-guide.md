## 目的

visions/ディレクトリ内の3つのサブディレクトリ（functions/, code-guides/, blueprints/）への文書配置基準と相互参照ルールを明確化し、適切な文書管理を実現する。

## 背景

- Architectによるvisions/構造の再編成により、specifications/→functions/への進化が実施された
- progress/ディレクトリは不要と判断され、進捗管理はAgent statusで実施
- KISS/YAGNI原則に基づき、シンプルで明確な3層構造を採用

## プロトコル詳細

### 1. ディレクトリ構造と責任範囲

#### functions/（機能仕様カタログ）
- **責任範囲**: 「何をするか」を定義
- **ファイル形式**: FUNC-XXX-{descriptive-name}.md
- **内容**:
  - 機能の目的と概要
  - ユーザー視点での振る舞い
  - 入出力の定義
  - 制約事項と前提条件
- **記載しないもの**: 実装詳細、技術的な実現方法

#### code-guides/（実装ガイド）
- **責任範囲**: 「どう作るか」を定義
- **ファイル形式**: CG-XXX-{implementation-pattern}.md
- **内容**:
  - 実装パターンとベストプラクティス
  - コーディング標準とスタイルガイド
  - 技術的な実装例
  - パフォーマンス考慮事項
- **記載しないもの**: 機能の目的、全体設計

#### blueprints/（統合設計図）
- **責任範囲**: 「全体像」を定義
- **ファイル形式**: BP-XXX-{system-design}.md
- **内容**:
  - システム全体のアーキテクチャ
  - コンポーネント間の関係
  - 複数機能の統合設計
  - 技術スタックの選定理由
- **記載しないもの**: 個別機能の詳細仕様、具体的なコード例

### 2. 文書作成時の判定基準

#### Q: 新しい文書をどのディレクトリに配置すべきか？

以下の質問に答えて判定：

1. **主な内容は何か？**
   - 機能の仕様・要件 → functions/
   - 実装方法・コードパターン → code-guides/
   - システム設計・アーキテクチャ → blueprints/

2. **誰が主な読者か？**
   - プロダクトオーナー・ユーザー → functions/
   - 開発者（実装担当） → code-guides/
   - アーキテクト・技術リード → blueprints/

3. **変更頻度は？**
   - 機能要件の変更時 → functions/
   - 実装技術の更新時 → code-guides/
   - アーキテクチャ変更時 → blueprints/

### 3. 相互参照のルール

#### 参照の方向性
```
blueprints/ → functions/ → code-guides/
     ↑              ↑             ↑
     └──────────────┴─────────────┘
           （逆参照も可能）
```

#### 参照形式
```markdown
## 関連文書
- 機能仕様: [FUNC-001-file-monitoring](../functions/FUNC-001-file-monitoring.md)
- 実装ガイド: [CG-001-event-handler-pattern](../code-guides/CG-001-event-handler-pattern.md)
- 統合設計: [BP-001-system-architecture](../blueprints/BP-001-system-architecture.md)
```

### 4. 番号採番ルール

#### 基本ルール
- 各ディレクトリ内で独立した通し番号
- 3桁のゼロパディング（001, 002, ...）
- 欠番は再利用しない

#### 採番例
- functions/: FUNC-001, FUNC-002, ...
- code-guides/: CG-001, CG-002, ...
- blueprints/: BP-001, BP-002, ...

### 5. 移行・再配置のガイドライン

#### 既存文書の再配置判定
1. 文書の主要内容を確認
2. 上記の判定基準に従って適切なディレクトリを決定
3. 必要に応じて内容を分割（機能仕様部分と実装部分を分離等）
4. 相互参照を追加して関連性を維持

#### 文書分割の例
```
旧: specifications/monitoring-system.md
  ↓
新: functions/FUNC-001-file-monitoring.md（機能仕様部分）
    code-guides/CG-001-event-handler-pattern.md（実装パターン部分）
    blueprints/BP-001-monitoring-architecture.md（全体設計部分）
```

### 6. 最新性維持原則

#### 基本方針
visions/ディレクトリは「生きたドキュメント」として、常に現在有効な情報のみを保持する。

#### 詳細ルール

1. **現在有効な情報のみ配置**
   - visions/内の全文書は最新かつ有効な仕様・設計・実装ガイドのみ
   - 実装済み機能も最新の仕様として維持
   - 将来の計画も現時点での最新版を反映

2. **歴史的資料の除外**
   - 廃止された設計書・仕様書は配置しない
   - 過去のバージョンの設計は配置しない
   - 採用されなかった提案・代替案は配置しない
   - **例外**: 移行作業中の参照文書（移行完了後は必ず削除）

3. **更新・削除の判断基準**
   - **更新対象**: 部分的に古くなったが、基本的に有効な文書
   - **削除対象**: 完全に無効化された文書、置き換えられた文書
   - **アーカイブ対象**: 歴史的価値はあるが現在無効な文書

4. **Git履歴での管理**
   - 削除された文書はGit履歴で参照可能
   - 重要な変更はコミットメッセージに記録
   - 大規模な更新時は移行記録を作成

5. **定期的なクリーンアップ**
   - 月次でvisions/内の文書の有効性を確認
   - 古くなった内容は即座に更新または削除
   - README.mdの更新を忘れずに実施

#### 実践例
```
# 良い例
functions/FUNC-001-file-monitoring.md（最新仕様）
blueprints/BP-002-for-version0200.md（現在の実装計画）

# 悪い例
functions/FUNC-001-file-monitoring-old.md（古いバージョン）
blueprints/BP-001-deprecated.md（廃止された設計）
functions/FUNC-001-file-monitoring-v1.md（バージョン付き）
```

## 実施手順

1. **新規文書作成時**:
   - 判定基準に従ってディレクトリを選択
   - 適切な番号を採番
   - 関連文書への相互参照を追加

2. **既存文書更新時**:
   - 配置の妥当性を再確認
   - 必要に応じて再配置・分割を検討
   - 相互参照を更新

3. **定期レビュー時**:
   - 各ディレクトリの文書配置を確認
   - 不適切な配置があれば移行計画を策定

## 成功基準

- [ ] 各文書が適切なディレクトリに配置されている
- [ ] 相互参照により関連文書が容易に発見できる
- [ ] 新規参加者が文書の場所を迷わず特定できる
- [ ] 重複内容が最小限に抑えられている
