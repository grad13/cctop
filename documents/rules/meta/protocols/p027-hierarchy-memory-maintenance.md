## 2. 階層定義と移行基準

### 2.1 L0: セッション記憶（揮発性）
- **内容**: 現在のClaude Codeセッション内の作業記憶
- **保持期間**: セッション終了まで
- **移行基準**: 重要情報は即座にL1へ記録

### 2.2 L1: status/{agent}.md（高頻度更新）
- **内容**: 進行中の作業、次回への引き継ぎ事項
- **更新頻度**: リアルタイム〜日次
- **保持期間**: 1週間程度
- **移行基準**: 
  - 作業完了時 → L2へ詳細化
  - 1週間以上更新なし → L2へ要約して移行

#### ステータス管理の具体的ルール（旧P009統合）
- **即座更新必須**: 30分以上の作業後は必ず更新
- **記録内容**: 現在の状態、今後の予定、ブロッカー
- **詳細移動**: 完了作業の詳細はrecords/reports/へ
- **一貫性維持**: 必ず前回の記載を確認してから更新

### 2.3 L2: records/（中頻度参照）
- **内容**: 完了作業の詳細記録、分析レポート
- **更新頻度**: 週次〜月次
- **保持期間**: 3ヶ月
- **移行基準**: 3ヶ月以上更新なし → L3へ

#### L2キャッシュ移行の具体的手順（旧P023統合）
**月次実施タスク**:
1. **候補特定**: 3ヶ月以上更新なしのrecords/reports/ファイル
2. **参照確認**: 他文書からの参照を確認
3. **移動実施**: documents/archives/に年月ディレクトリ作成して移動
4. **リンク更新**: 移動したファイルを参照しているリンクを修正

### 2.4 L3: documents/archives/（長期保存）
- **内容**: 歴史的記録、参照専用情報
- **更新頻度**: 原則として更新しない（Read-only）
- **保持期間**: 無期限
- **アクセス**: Clerk管理、他Agent読取専用

## 3. records/配下の配置基準

### 3.1 配置判断フローチャート

```
記録すべき内容がある
    ↓
バグに関する内容か？
    ├─ Yes → 未解決: records/bugs/ （フラット構造）
    │        解決済: documents/archives/bugs/ へ即座移動
    │
    └─ No → 分析・設計・実装の記録か？
             ├─ Yes → records/reports/
             │        （1レポート1トピック厳守）
             │
             └─ No → インシデントか？
                      ├─ Yes → records/incidents/
                      └─ No → Clerkに相談
```

### 3.2 具体的な配置例

#### records/bugs/配下に配置すべきもの（2025年6月16日更新）
- **アクティブなバグ**: 直接 `records/bugs/` に配置（フラット構造）
- 再現手順・原因分析
- 修正内容・テスト結果
- **解決済みバグ**: 即座に `documents/archives/bugs/` へ移動
- **注**: active/archiveサブディレクトリは2025年6月16日に廃止

#### records/reports/配下に配置すべきもの
- 機能実装の詳細レポート
- システム分析結果
- 設計書・仕様書の草案（確定版はroadmaps/specifications/へ移行）
- 作業完了報告（バグ以外）
- 統計・監視レポート

#### records/incidents/配下に配置すべきもの
- プロセス違反の記録
- システム障害の記録
- 緊急対応の記録

## 4. 1レポート1トピック原則の徹底

**適用範囲**: この原則は`records/`配下のすべてのサブディレクトリに適用されます
- `records/reports/` - 分析・設計・実装レポート
- `records/bugs/` - バグ記録
- `records/experiments/` - 実験記録
- `records/incidents/` - インシデント記録

### 4.1 違反例（避けるべき）
```markdown
# REP-XXXX: 2025年6月14日〜15日の作業まとめ
- TimeBox修正について
- Vision同期の実装について  
- エスカレーション対応について
```
→ 複数のトピックが混在

### 4.2 正しい例
```markdown
# REP-XXXX: TimeBox状態管理修正
（TimeBoxの修正のみに焦点）

# REP-YYYY: Vision同期機能実装
（Vision同期のみに焦点）

# REP-ZZZZ: エスカレーション対応プロセス改善
（エスカレーションのみに焦点）
```

### 4.3 既存レポートの扱い
- **REP-0031のような複合レポート**: そのまま保持（歴史的記録）
- **今後の作成**: 必ず1レポート1トピックを遵守
- **分割が必要な場合**: 新規レポートとして作成し、相互参照

### 4.4 各ディレクトリでの適用例
- **reports/**: 1つの分析・設計・実装に焦点
- **bugs/**: 1つのバグ現象と修正に焦点
- **experiments/**: 1つの実験仮説と結果に焦点
- **incidents/**: 1つのインシデントと対応に焦点

## 5. 階層メンテナンス手順

### 5.1 日次メンテナンス
1. **L0→L1**: セッション終了前にstatus更新
2. **L1整理**: 不要な情報を削除、重要情報を保持

### 5.2 週次メンテナンス
1. **L1→L2**: 完了作業を詳細レポート化
2. **配置確認**: bugs/reports/incidents/の適切な振り分け
3. **1トピック確認**: 複数トピックは分割

### 5.3 月次メンテナンス（P024準拠）
1. **L2レビュー**: 3ヶ月経過した文書の確認
2. **L2→L3**: 該当文書をarchive/へ移動
3. **整合性確認**: 参照リンクの更新
