# P024: 月次文書管理レビュープロトコル

**プロトコルID**: P024  
**作成日**: 2025年6月16日 nodata  
**作成者**: Clerk Agent  
**カテゴリ**: 文書管理  
**実行頻度**: 月次（毎月第1営業日）  
**関連仮説**: 旧H045（階層的文書ライフサイクル管理）統合済み  

## 📋 概要

月次で文書の健全性を確認し、適切な階層（L1/L2/L3）への移行を促進する総合的なレビュープロトコル。P022のような orchestration protocol として、複数のサブプロトコルを統合的に実行する。

## 🎯 目的

1. **文書鮮度の維持**: 各層に適切な文書のみを保持
2. **ライフサイクル管理**: 文書の誕生から保存までを体系的に管理
3. **品質保証**: 文書構造の健全性を定期的に確認
4. **改善機会の発見**: 運用上の問題を早期に検出

### 文書ライフサイクルの定義（H045由来）

```
誕生 → 成長 → 成熟 → 保存
L0   →  L1  →  L2  →  L3

L0: セッション中の作業（揮発性）
L1: 進行中の記録（日次更新）status/{agent}.md
L2: 完了した重要記録（週次～月次）records/reports/
L3: 歴史的アーカイブ（年次）documents/archives/
```

## 📊 実行タイミング

- **定期実行**: 毎月第1営業日の午前中
- **推奨時刻**: 10:00-11:00（他の作業が少ない時間帯）
- **所要時間**: 30-45分（初回は60分程度）

## 🔄 実行手順

### Phase 1: 準備（5分）

```bash
# 1. 作業開始をstatus/clerk.mdに記録
echo "月次文書管理レビュー開始: $(date)" >> status/clerk.md

# 2. 前回レビューの結果確認
cat records/reports/REP-*-monthly-review-*.md | tail -1
```

### Phase 2: L2キャッシュ確認（15分）

```bash
# 1. P023実行（L2→L3移行チェック）
# 詳細はP023プロトコル参照
```

チェック項目:
- [ ] 3ヶ月経過ファイルの特定
- [ ] 完了済み計画書の確認
- [ ] 移行候補リストの作成
- [ ] P023に基づく移行実施

#### L2→L3移行基準（H045由来）
- **計画書**: 実施完了から1週間
- **作業記録**: 作成から3ヶ月経過
- **定期記録**: 更新停止から3ヶ月
- **例外**: 継続更新型（P007履歴等）は移行しない

### Phase 3: L1ステータス確認（10分）

#### L1→L2移行基準（H045由来）
- 作業の完了（マイルストーン達成）
- 他エージェントとの共有必要性
- 2週間以上の安定性（頻繁な更新終了）

```bash
# 1. 各エージェントのstatus確認
ls -la documents/agents/status/*.md

# 2. 肥大化チェック（100行超）
wc -l documents/agents/status/*.md
```

チェック項目:
- [ ] 2週間以上更新されていない項目
- [ ] L2への昇格候補の特定
- [ ] 不要な記録の削除候補

## 🚨 注意事項

### やってはいけないこと
- ❌ 確認なしの大量移行
- ❌ 番号体系の変更
- ❌ 継続更新型ファイルの移行
- ❌ 権限外ディレクトリの操作

### 判断に迷った場合
1. 移行を保留してレポートに記載
2. ユーザーに確認を求める
3. 次回レビューで再検討

## 📊 メトリクス

### 収集項目
- L2ファイル総数と3ヶ月以内の割合
- L1の平均行数
- 移行作業の所要時間
- 発見した問題の数と種類

### 目標値（H045由来の成功指標）
- L2の鮮度: 80%以上が3ヶ月以内
- レビュー時間: 30分以内
- 誤移行: ゼロ
- 検索効率: 必要文書の発見時間50%短縮

## 🔄 改善プロセス

問題や改善案を発見した場合:
1. レビューレポートに記載
2. 重大な問題はインシデント作成
3. 改善サイクルデータとして活用
4. 必要に応じてP023/P024を更新
