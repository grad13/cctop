## 📋 概要

プロトコルの内容を定期的に精査し、参照の正確性・内容の最新性・他プロトコルとの整合性を確保するための包括的見直しプロセス。patternsコマンドでの自動検出には限界があるため、Clerk Agentによる包括的精読と体系的な改修計画策定を基本とする。

## 🎯 目的

1. **参照の正確性確保**: 移動・削除されたファイルへの参照を修正
2. **内容の最新性維持**: 古い情報・廃止された手順の更新
3. **整合性の確保**: プロトコル間の矛盾・重複の解消
4. **品質の向上**: 実運用で判明した改善点の反映

## ⏰ 実施頻度とタイミング

### 定期見直し
- **月次**: 全プロトコルの包括的見直し（毎月第1週）
- **随時**: 大きな変更（ディレクトリ移動、体制変更等）発生時

### 実施タイミングの想定
- **夜間依頼**: 寝る前に依頼し、時間をかけて精読・分析
- **無時間制約**: 品質を優先し、十分な時間を確保
- **2段階プロセス**: 改修計画作成→実施の分離で確実性向上

### トリガー条件
以下の場合は定期見直しとは別に即座実施：
- ディレクトリ構造の大幅変更
- エージェント体制の変更
- 新規プロトコルの大量追加（5個以上）
- ユーザーからの整合性指摘

## 📋 見直し対象と観点

### 対象範囲
- `documents/rules/meta/protocols/`内の全.mdファイル
- P000～P999の全番号（欠番含む）
- protocols/README.mdの整合性

### 4つの見直し観点

#### 1. 参照の正確性
```markdown
チェック項目：
- [ ] ファイルパスの存在確認
- [ ] 他プロトコルへの参照番号の正確性
- [ ] 外部文書（specifications/、roadmaps/等）への参照
- [ ] CLAUDE.md内の参照との整合性
```

#### 2. 内容の最新性
```markdown
チェック項目：
- [ ] 廃止されたディレクトリ（daily/、hypotheses/等）への言及
- [ ] 古いエージェント名（Monitor→Inspector等）
- [ ] 変更された手順・ルールの更新
- [ ] 新機能・新体制への対応状況
```

#### 3. プロトコル間整合性
```markdown
チェック項目：
- [ ] 重複する内容の統合可能性
- [ ] 矛盾する記述の調整
- [ ] 関連プロトコル間の相互参照
- [ ] 階層関係の適切性
```

#### 4. 実用性と改善
```markdown
チェック項目：
- [ ] 実運用での有効性
- [ ] 手順の具体性・実行可能性
- [ ] チェックリストの完全性
- [ ] 例外ケースの網羅性
```

## 🔄 実施プロセス（2段階プロセス）

### 第1段階: 包括的精読と改修計画作成
**想定時間**: 3-5時間（時間制約なし）

#### Step 1: 現状把握（20分）
1. **プロトコル一覧作成**
   ```bash
   find documents/rules/meta/protocols -name "p*.md" | sort -V
   ```
2. **protocols/README.md精査**
   - 番号体系と実ファイルの一致性
   - 欠番管理の適切性
3. **前回以降の変更履歴確認**
   - git log --since="1 month ago" --oneline
   - 主要変更のインパクト分析

#### Step 2: 全プロトコル精読（2-3時間）
各プロトコルについて以下を精査：

##### 2.1 基本情報チェック
- [ ] Protocol IDとファイル名の一致
- [ ] メタデータの完全性（作成日、更新日、ステータス）
- [ ] カテゴリ分類の適切性
- [ ] 疑問点・決定事項セクションの完了状況

##### 2.2 内容品質精査
- [ ] **参照の正確性**: ファイルパス・他プロトコル番号の存在確認
- [ ] **内容の最新性**: 廃止ディレクトリ・古い手順・旧エージェント名への言及
- [ ] **プロトコル間整合性**: 重複・矛盾・相互参照の適切性
- [ ] **実用性**: 手順の具体性・実行可能性・例外ケースの網羅

##### 2.3 問題分類と記録
各プロトコルを以下に分類：
- **Critical**: 致命的問題（リンク切れ、重大矛盾）
- **Important**: 重要問題（古い情報、整合性不備）
- **Nice-to-have**: 改善推奨（説明充実、例示追加）
- **Consolidation**: 統廃合検討（重複・分割統合）

#### Step 3: 改修計画策定（30-60分）
1. **問題の優先度付け**
   - Critical: 即座対応必要
   - Important: 今回対応推奨
   - Nice-to-have: 時間があれば対応
   - Consolidation: 別途計画策定

2. **詳細作業計画作成**
   - 各プロトコルの具体的更新内容
   - 作業所要時間の見積もり
   - 関連文書への影響範囲

3. **改修計画レポート作成**
   - `records/reports/REP-XXXX-protocol-review-plan-YYYY-MM.md`作成
   - 発見した問題の詳細、改修方針、作業順序を記録

---

### 第2段階: 改修計画に基づく実施
**想定時間**: 1-3時間（問題数により変動）

#### Step 4: 計画的更新実施
1. **改修計画レポート確認**
   - 前段階で作成した計画の再確認
   - 優先度順での作業順序決定

2. **プロトコルファイル更新**
   - Critical/Important問題を優先実施
   - 各ファイルの更新履歴記録
   - 最終更新日の更新

3. **関連文書の一括更新**
   - protocols/README.mdの整合性修正
   - CLAUDE.md内参照の更新（必要時）
   - 他プロトコルからの相互参照更新

#### Step 5: 結果検証と記録
1. **品質検証**
   - リンク切れの全面チェック
   - 参照の正確性再確認
   - 更新内容の整合性確認

2. **実施結果レポート作成**
   - `records/reports/REP-XXXX-protocol-review-result-YYYY-MM.md`作成
   - 実施内容の詳細、成果、次回への申し送り事項を記録

## ℹ️ 安全性と品質保証

### 更新時の必須確認
1. **波及影響分析**: 更新による他プロトコル・CLAUDE.mdへの影響
2. **実運用インパクト**: 現在使用中のプロセスへの影響範囲
3. **ロールバック計画**: 問題発生時の復旧手順

### 品質管理措置
- **Git履歴管理**: 更新前に必ずcommitで安全な状態を保存
- **段階的更新**: 大幅変更は小分けして段階的に実施
- **ユーザー確認**: 統廃合や大幅変更は事前相談を実施
- **実績ベース判断**: 実際の利用状況を踏まえた慎重な統廃合判断
