# P044: L1→L2移行プロトコル

**プロトコルID**: P044  
**制定日**: 2025年6月19日  
**更新日**: 2025年6月19日（内容精査ルール追加）  
**制定者**: Clerk  
**状態**: 改訂済み  
**カテゴリー**: 文書管理  
**関連**: DDD2（階層メモリメンテナンス原則）、REP-0084

## 目的

status/{agent}.md（L1）から records/（L2）への週次移行を標準化し、statusファイルの肥大化を防止しつつ、完了作業の知識を体系的に保存する。

## 背景

- **roles/status分離**: 2025年6月19日実施により、statusは純粋な進捗記録に特化
- **問題**: L1→L2移行が不定期・不統一で、エージェント性能低下の原因に
- **解決**: 機械的・定期的な移行プロセスの確立

## 基本原則

### 1. 3日間 + 内容精査ルール
- **3日経過**: 機械的判定による自動移行（判断不要）
- **3日未満**: 内容精査による早期移行判定
- **容量基準**: 500行超過時は期間に関係なく実施

### 2. 内容理解による分類
- statusの記録内容を実際に読み、適切なrecords/サブディレクトリに分類
- 機械的分類ではなく、文脈理解に基づく判断

### 3. 自己完結性
- 各エージェントが自身のstatusを自己管理
- records/配下は全エージェント編集可能なので権限問題なし

### 4. 柔軟な実施タイミング
- 定期実施より、必要に応じた実施を重視
- ユーザー依頼時の実施も想定

## 移行基準

### 基本判定ルール
1. **3日経過ルール（機械的判定）**
   - 作成から3日（72時間）経過 → 自動的に移行対象
   - 判断不要・機械的に移行実行

2. **容量基準ルール（強制実行）**
   - statusファイルが500行超過 → 期間に関係なく移行実施
   - 古いものから優先的に移行

3. **3日未満ルール（内容精査必須）**
   - 作成から3日未満でも内容精査により早期移行可能
   - **禁止**: 機械的・自動的な判定
   - **必須**: 人間による内容確認・判断

### 早期移行判定基準（3日未満）

#### 移行推奨条件
- **完了作業**: 明確に完了・解決済みの作業
- **一時記録**: 特定の作業履歴で継続参照不要
- **文書化済み**: REP/BUG/INC番号が付与されて体系化済み
- **実験終了**: 試行・検証の完了記録

#### 移行禁止条件（L1保持必須）
- **進行中作業**: 「進行中」「作業中」「検討中」等
- **継続責務**: 「監視」「メンテナンス」「定期確認」等
- **次回予定**: まだ着手していないタスク
- **頻繁参照**: 現在の作業で頻繁に参照している情報

#### 慎重判断条件（要検討）
- **境界事例**: 移行・保持の判断が微妙
- **関連作業**: 現在進行中の作業と間接的に関連
- **デフォルト判定**: 迷った場合はrecords/reports/へ移行

## 実施手順

### 1. 実施タイミング
- **3日経過チェック**: 適宜実施（週1-2回程度）
- **容量基準**: statusが500行超過時は即座実施
- **ユーザー依頼**: 依頼があった場合は優先実施
- **所要時間**: 1件あたり3-5分の内容精査時間を確保

### 2. 手順

#### Step 1: 対象識別・内容精査
**3日経過ファイルの機械的抽出**:
```bash
# 自身のstatusファイルを確認
Read documents/agents/status/{agent}.md

# 3日以上前の日付を探す（機械的判定）
# 容量確認: 500行超過かチェック
```

**3日未満ファイルの内容精査**:
- **全文読了**: 該当箇所の内容を詳細に確認
- **作業背景理解**: なぜその作業が必要だったかを理解
- **現在の関連性**: 現在進行中の作業との関連を確認
- **完了状況**: 作業が本当に完了しているかを判断
- **参照頻度**: 今後参照する可能性を検討

#### Step 2: 移行先決定（内容理解による分類）

**分類基準**:
- **bugs**: DDD0の最も抽象レベルの低い問題（具体的なコード・機能の不具合）
- **incidents**: DDD0のmeta以上の層での違反（プロセス・ルール・システム違反）
- **reports**: 上記以外の全て（実装・分析・設計・調査・完了記録等）

**判定プロセス**:
1. 具体的なバグ・不具合か？ → records/bugs/
2. メタレベル・プロセス違反か？ → records/incidents/
3. その他・判断に迷う場合 → records/reports/（デフォルト）

#### Step 3: 移行実施（3分）
```bash
# A. 新規ファイル作成の場合
Write documents/records/reports/REP-XXXX-{brief-title}-{date}.md
- 内容をそのままコピー
- 最小限のヘッダー追加

# B. 既存ファイルに追記の場合（関連する内容）
Edit documents/records/reports/REP-XXXX-{existing}.md
- 日付セクションを追加
- 内容を追記
```

#### Step 4: status更新（1分）
```bash
# statusから該当部分を削除
Edit documents/agents/status/{agent}.md
- 移行した内容を削除
- 必要なら「詳細はREP-XXXXを参照」を残す
```

#### Step 5: README更新（2分）
```bash
# records/reports/README.mdを更新（新規作成時のみ）
Edit documents/records/reports/README.md
- 新規レポートをテーブルに追加
```

#### Step 6: 移行完了確認
```bash
# 移行内容の確認
- 適切なrecords/サブディレクトリに配置されているか
- statusから該当部分が削除されているか
- README.md更新が必要な場合は実施済みか
```

## 品質基準（最小限）

### 必須要素
- **日付**: いつの作業か分かること
- **内容**: 何をしたか分かること
- **結果**: どうなったか分かること

### 推奨要素（時間があれば）
- **背景**: なぜ必要だったか
- **手順**: どうやったか
- **学び**: 次回への知見

### 不要な要素
- 完璧な文章構成
- 詳細な技術説明（statusのままでOK）
- 他者向けの丁寧な説明

## 例外処理

### 判断に迷う場合
- **デフォルト**: records/reports/に移行
- **原則**: 迷ったら移行する（statusに残さない）

### 大量の移行対象がある場合
- **分割実施**: 最大30分で打ち切り、次週に継続
- **優先順位**: 古いものから順に

### 関連性の高い内容
- **統合OK**: 1つのレポートにまとめて問題なし
- **判断基準**: 同じ週の同じトピックなら統合

## 効果測定

### 成功指標
- statusファイルが500行以内に維持される（強制実行基準）
- 3日経過分は確実に移行される
- 内容精査による適切な分類ができている
- records/内の文書が適切に分類されている

### 失敗指標
- statusファイルが500行を頻繁に超過する
- 機械的判定による誤分類が発生
- 内容精査を怠った不適切な移行
