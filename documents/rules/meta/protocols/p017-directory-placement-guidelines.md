## 🎯 目的

文書の配置先を明確化し、記録系ディレクトリでの重複許容と体系系ディレクトリでのMECE（Mutually Exclusive, Collectively Exhaustive）原則を両立する。H030から統合されたディレクトリ役割分担ルールとH043のMECE原則により、「現在の状態」「将来の計画」「開発ルール」「作業記録」「過去の資料」の明確な区別を実現する。

## 📊 ディレクトリ分類と配置原則

### MECE原則の適用（H043由来）

**記録系ディレクトリ（status, incidents, bugs, archive, experiments, reports）では重複を許容し、それ以外のディレクトリではMECE原則を厳守することで、文書管理の柔軟性と体系性を両立する。**

### 1. 記録系ディレクトリ（重複許容）

| ディレクトリ | 用途 | 配置するもの | 重複方針 |
|-------------|------|------------|----------|
| **status/** | 現在状況 | ・各Agentの現在作業<br>・進行中タスク<br>・次の予定 | 最新のみ保持<br>過去はreportsへ |
| **incidents/** | 問題記録 | ・インシデント詳細<br>・5フェーズ対応記録<br>・根本原因分析 | 1インシデント1ファイル<br>関連情報は参照 |
| **bugs/** | バグ追跡 | ・バグ詳細記録<br>・修正過程<br>・テンプレート | active/archive分離<br>状態遷移を記録 |
| **archive/** | 過去文書 | ・廃止された文書<br>・過去バージョン<br>・歴史的記録<br>・完了マイルストーン | 完全許容<br>削除より保存<br>YYYY-MM/形式 |
| **experiments/** | 実験記録 | ・試行錯誤の過程<br>・実験結果<br>・失敗記録 | 完全許容<br>過程を全記録 |
| **reports/** | 実施記録 | ・定期チェック結果<br>・分析レポート<br>・活動履歴 | 時系列保存<br>傾向分析用 |

### 2. 体系系ディレクトリ（MECE原則）

| ディレクトリ | 用途 | 配置するもの | MECE方針 |
|-------------|------|------------|----------|
| **specifications/** | 仕様書 | ・機能仕様<br>・技術仕様<br>・インターフェース仕様<br>・実装済みアーキテクチャ<br>・確定したビジネス戦略 | 完全MECE<br>1機能1文書<br>現在の状態説明 |
| **protocols/** | プロトコル | ・開発ルール<br>・作業手順<br>・ガイドライン | 完全MECE<br>1プロトコル1文書<br>統合時は明記 |
| **protocols/** | 手順書 | ・作業手順<br>・ルール<br>・ガイドライン | 完全MECE<br>1手順1文書<br>バージョン管理 |
| **roadmaps/** | 計画 | ・プロジェクト計画<br>・マイルストーン<br>・スケジュール<br>・機能別開発計画・議論 | 完全MECE<br>1計画1文書<br>将来の実装 |
| **checklists/** | チェックリスト | ・確認項目<br>・手順リスト | 完全MECE<br>1目的1リスト<br>参照専用 |
| **bugs/** | バグ記録 | ・バグ詳細<br>・修正記録 | 完全MECE<br>1バグ1文書<br>active/archive分離 |

### 3. 特殊ディレクトリ

| ディレクトリ | 用途 | 配置するもの | 特記事項 |
|-------------|------|------------|----------|
| **dominants/** | 最上位規則 | ・不変の原則<br>・最高優先度ルール | 完全MECE<br>極少数精鋭<br>重複厳禁 |
| **directions/** | 確立手法 | ・検証済み手順<br>・ベストプラクティス | 完全MECE<br>1手法1文書<br>昇格のみ |
| **meta/** | メタ管理 | ・下位ディレクトリで個別管理 | 下位ごとに分類適用 |

## 🔄 情報の流れとライフサイクル

### 典型的な流れ

```
1. 作業開始
   ↓
2. status/{agent}.mdに「現在の作業」として記録
   ↓
3. 作業中の詳細はstatusに追記
   ↓
4. 作業完了後、詳細をrecords/reports/REP-XXXX-*.mdに移動
   ↓
5. statusには要約と「→ records/reports/REP-XXXX-*.md参照」を残す
   ↓
6. 重要な成果物は該当する体系系ディレクトリに配置
   ↓
7. 1週間後、statusから過去の要約も削除（reportsに完全移行）
```

### 情報の重複管理

#### 許可される重複
- reports内での同一情報の複数記載
- statusからreportsへの移行期間中の一時的重複
- incidentでの経緯説明における部分的重複
- bugs内でのactive→archive移行時の履歴保持
- experiments内での試行錯誤の全過程記録
- reports内での定期レポートの蓄積

#### 禁止される重複（MECE違反）
- specifications内での同一仕様の複数文書化
- protocols内での同一手順の重複記載  
- hypotheses内での同一仮説の複数管理
- dominants内での概念の重複
- directions内での手法の重複
- roadmaps内での計画の重複
- checklists内での確認項目の重複

#### 体系系でのMECEルール（H043由来）
1. **単一責任**: 1文書1トピック
2. **相互排他**: 内容の重複禁止
3. **完全網羅**: カテゴリー内で漏れなし
4. **参照活用**: 関連情報は参照リンクで接続

## 📝 配置判断フローチャート

```
新規文書を作成したい
    ↓
Q1: 時系列の記録か？
    Yes → records/reports/へ
    No → Q2へ
    ↓
Q2: 現在進行中の作業記録か？
    Yes → status/へ
    No → Q3へ
    ↓
Q3: 問題・エラー・違反の記録か？
    Yes → incidents/（またはbugs/）へ
    No → Q4へ
    ↓
Q4: 仕様・手順・計画等の正式文書か？
    Yes → 該当する体系系ディレクトリへ（MECE確認）
    No → meta/またはその他適切な場所へ
```

## ⚠️ よくある配置ミス

### ❌ 避けるべきパターン
1. **仕様書の散在**: 同じ機能の仕様がspecificationsとroadmapsに分散
2. **手順の重複**: 同じ作業手順がprotocolsとchecklistsに存在
3. **古い情報の放置**: statusに1週間以上前の情報が残存

### ✅ 正しいパターン
1. **仕様の一元化**: specifications/に集約、他は参照のみ
2. **手順の分離**: protocols（詳細手順）、checklists（確認項目）
3. **定期的な移行**: status → reports → （必要なら）archive

## 🔍 MECE確認チェックリスト

## 🔢 記録系ディレクトリの番号体系

### 各ディレクトリの命名規則

| ディレクトリ | プレフィックス | 番号体系 | 例 |
|-------------|--------------|----------|----|
| **experiments/** | EXP- | YYYYMMDD-XXX | `EXP-20250615-001-records-performance-test.md` |
| **reports/** | REP- | XXXX（4桁） | `REP-0001-meta-reorganization-plan.md` |
| **bugs/** | BUG- | YYYYMMDD-XXX | `BUG-20250615-001-timebox-timer-abnormal.md` |
| **incidents/** | INC- | YYYYMMDD-XXX | `INC-20250615-001-coder-unauthorized-resolution.md` |

### 番号管理の原則

1. **連番管理**: 各ディレクトリで独立した通し番号
2. **欠番禁止**: 削除・統合時も番号は再利用しない
3. **README更新**: 新規作成時は必ずREADME.mdの一覧表を更新
4. **参照簡素化**: 「EXP-001を参照」のような短い表記で参照可能

### 日付形式
- **YYYYMMDD**: ハイフンなしの8桁（20250615）
- **XXX**: 3桁のゼロ埋め（001, 002, ..., 999）
- **XXXX**: 4桁のゼロ埋め（0001, 0002, ..., 9999）

## ⛔ 禁止事項（H030由来）

### 重複概念のディレクトリ作成禁止
- `questions/` と `roadmaps/` の重複
- `completed/` と `archive/` の重複
- `policies/` と `rules/` の重複

### 曖昧な配置の禁止
- 「どちらにも該当しそう」な場合は必ずユーザーに確認

### 命名規則違反の禁止
- 必ず `YYYY-MM-DD-HHMM-description.md` 形式を使用（記録系）
