# P043: L2→L3アーカイブ移行プロトコル

**プロトコル番号**: P043  
**作成日**: 2025年6月18日  
**策定者**: Clerk Agent  
**適用範囲**: documents/records/配下の全文書  
**関連**: DDD2階層メモリメンテナンス原則、REP-0079  

## 📋 プロトコル概要

### 目的
DDD2階層メンテナンス原則に基づくL2（records/）からL3（archives/）への文書移行を体系化し、検索継続性とtraceabilityを保持する。

### 基本原則
1. **2週間 + 内容精査ルール**: 機械的時間基準と人間判断の併用
2. **patterns検索継続性**: 移行後もキーワード検索可能性を保証  
3. **完全traceability**: 移行履歴と参照関係の完全保持
4. **機械的判定禁止**: 2週間未満での自動・機械的アーカイブ判定を禁止

## ⚠️ 移行判断基準

### 基本判定ルール
1. **2週間経過ルール（機械的判定）**
   - 作成から2週間（14日）経過 → 自動的にアーカイブ対象
   - 判断不要・機械的に移行実行

2. **2週間未満ルール（内容精査必須）**
   - 作成から2週間未満でも内容精査により早期アーカイブ可能
   - **禁止**: 機械的・自動的な判定

### 早期アーカイブ判定基準（2週間未満）

#### アーカイブ推奨条件
- **実行完了計画**: 計画が完全実行済みで参照価値低下
- **一時的記録**: 特定の作業履歴で継続参照不要
- **重複解消**: 他文書で同内容が体系化済み
- **実験終了**: 試行・検証の完了記録

#### アーカイブ禁止条件（絶対保持）
- **継続参照**: 他文書から頻繁に参照されている
- **進行中関連**: 関連プロジェクト・タスクが継続中
- **技術基盤**: システム・プロセスの基盤文書
- **最新仕様**: 現在有効な仕様・設計書
- **教訓価値**: 将来の参考価値が高い失敗・成功事例

#### 慎重判断条件（要検討）
- **境界事例**: アーカイブ・保持の判断が微妙
- **影響不明**: アーカイブ後の影響範囲が不明確
- **複合要素**: 複数の判定要素が混在

## 🔍 移行プロセス

### Phase 1: 移行候補抽出・判定

#### 2週間経過ファイルの処理
1. **機械的抽出**: `find documents/records/ -name "*.md" -mtime +14`
2. **自動移行**: 判断不要で即座にアーカイブ実行
3. **移行記録**: 機械的移行の一括記録

#### 2週間未満ファイルの処理
1. **候補選定**: 早期アーカイブを検討すべきファイルの人間による選定
2. **内容精査プロセス**: 以下の手順で詳細確認
   - **ファイル全文読了**: タイトルやメタデータのみでなく、内容全体を確認
   - **作成背景理解**: なぜ作成されたか、何の問題を解決したかを理解
   - **現在の関連性**: 現在進行中のプロジェクト・タスクとの関連を確認
   - **将来の参照価値**: 今後参照される可能性を検討
   - **代替情報源**: 同じ内容が他の文書で体系化されているか確認
3. **多角的判断**: 以下の観点から総合判定
   - **実用性**: 現在・将来の実用価値
   - **独自性**: 他では得られない情報の有無
   - **完了性**: 関連作業・プロジェクトの完了状況
   - **教訓価値**: 失敗・成功事例としての学習価値
4. **判断記録**: 各判定の理由・根拠・考慮した要素を詳細に記録

### Phase 2: 移行前準備
1. **参照確認**: 他文書からのリンク・参照の全検索
2. **キーワード抽出**: patterns検索用キーワードの明確化（10-20個選定）
3. **移行記録**: 移行理由・参照情報の記録

### Phase 3: 実移行実行
1. **キーワード追加**: ファイル先頭にアーカイブ情報・検索キーワード挿入（10-20個）
2. **ファイル移動とリネーム**: 
   - documents/archives/YYYY/MMDD-MMDD/配下へ移動
   - ファイル名を`YYYY-MM-DD-HHMM-適切な内容説明.md`形式に変更
   - YYYY-MM-DD-HHMMは元ファイルの作成日時
   - 内容説明部分は文書の内容を反映した適切なkebab-case名
3. **リンク更新**: 参照している他文書のリンク修正
4. **README一覧削除**: アーカイブされたファイルをREADME.mdのテーブルから完全削除

### Phase 4: 移行後確認
1. **patterns検索テスト**: 追加キーワードでのarchives/内検索確認
2. **参照確認**: リンク切れの全確認
3. **記録更新**: 移行完了の記録・統計更新
4. **番号管理**: README.mdでは欠番が生じるが、新規レポートは常にincrementalに次の番号を使用

## 📁 archives/内構造設計

### 基本ディレクトリ構造（週次・月曜開始）
```
documents/archives/
├── 2025/0616-0622/  # 2025年6月16日週（月曜開始）
├── 2025/0609-0615/  # 2025年6月9日週  
├── 2025/0602-0608/  # 2025年6月2日週
└── 2024/1216-1222/  # 過去分
```

### 分類基準
- **週次ベース**: アーカイブ実行週による分類のみ（月曜開始）
- **フラット構造**: 週別ディレクトリ内はフラット配置
- **内容分類なし**: 機能・内容による分類は行わない
- **ディレクトリ命名**: YYYY/MMDD-MMDD形式（開始月曜-終了日曜）

### アーカイブファイル命名規則
- **形式**: `YYYY-MM-DD-HHMM-内容説明.md`
- **YYYY-MM-DD-HHMM**: 元ファイルの作成日時（作成日が不明な場合は最終更新日時）
- **内容説明**: 文書の内容を表す適切なkebab-case名
- **例**: 
  - `2025-06-15-1430-five-agent-system-migration.md`
  - `2025-06-18-0900-ddd2-hierarchy-implementation.md`
- **理由**: ディレクトリベースではなくキーワードベースの管理を前提とするため

## 🔍 patterns検索継続性保証

### キーワード追加戦略
1. **ファイル内キーワード追加**: アーカイブ時にファイル先頭または末尾にキーワード挿入（5-10個）
2. **週次ベース配置**: archives/YYYY/MMDD-MMDD/配下にシンプル配置
3. **patterns検索活用**: キーワードでarchives/全体を検索
4. **検索ヒット率向上**: 複数キーワードによる発見可能性の最大化

### アーカイブファイル形式
```markdown
# 元ファイル: REP-0079-reports-readme-optimization-strategy-20250618.md

アーカイブ時に以下を追加:

---
**アーカイブ情報**
- アーカイブ日: 2025-07-21（月曜日）
- アーカイブ週: 2025/0721-0727
- 元パス: documents/records/reports/
- 検索キーワード: README最適化, DDD2, patterns戦略, 軽量化, Web標準, 文書管理, 階層メンテナンス, アーカイブ移行

---

[元の内容そのまま]
```

### patterns検索例
```bash
# アーカイブ内検索
patterns "DDD2" archives/                    # 全アーカイブ検索
patterns "README最適化" archives/             # 特定キーワード検索  
patterns "SQLite" archives/2025/0616-0622/   # 特定週のアーカイブ検索
patterns "UI改善" archives/2025/06*/          # 6月全週検索
```

## 📋 README.md管理ルール

### アーカイブ時のREADME更新
1. **テーブル行削除**: アーカイブされたレポート行をREADME.mdのテーブルから完全削除
2. **アーカイブ記録**: 上部の「DDD2実行」セクションにアーカイブ履歴を追記
3. **検索案内**: アーカイブ済みファイルのキーワード検索方法を案内

### 番号管理原則
- **欠番許容**: アーカイブによりREP番号に欠番が生じるが、これは正常動作
- **incremental採番**: 新規レポートは欠番を埋めず、常に最新番号+1を使用
- **例**: REP-0019がアーカイブされても、次は REP-0080, REP-0081... と継続
- **検索継続**: アーカイブされたレポートはpatterns検索で発見可能

### README.mdテーブルの目的
- **現行レポート**: 現在参照可能・編集可能なレポート一覧
- **アーカイブ分離**: archives/配下の情報は別途管理
- **軽量維持**: テーブルサイズの肥大化防止

## 📊 移行統計・監視

### 必須記録項目
- **移行日時**: 実行日・時刻
- **移行理由**: 判断基準・経過期間
- **移行先**: archives/内の具体的パス
- **キーワード**: 主要検索キーワード（5-10個）
- **参照影響**: 影響を受けた他文書

### 定期レビュー
- **週次**: 2週間経過ファイルの機械的移行 + 内容精査による早期移行判定
- **月次**: 移行効果・検索効率評価・内容精査精度の見直し
- **四半期**: archives/構造・分類基準・判定基準の見直し

## ⚠️ 緊急時対応

### 誤移行時の復旧
1. **即座復旧**: archives/からrecords/への移動
2. **リンク修復**: 参照関係の即座復元
3. **記録更新**: 誤移行の原因・対策記録

### 検索不能時の対応
1. **キーワード追加**: 不足キーワードの補完
2. **構造見直し**: archives/分類の調整
3. **ガイド更新**: 検索方法の改善

## 🎯 実装スケジュール

### 改訂実行（2025年6月19日）
- **判定基準改訂完了**（2週間 + 内容精査ルール）
- **機械的判定禁止原則の確立**（2週間未満での機械的アーカイブ禁止）
- **内容精査ベースの移行プロセス確立**

#### 改訂の背景・教訓
1. **機械的移行の問題**：
   - 2025年6月19日、REP-0020〜0086の65件を機械的に移行しようとした
   - ユーザーより「内容をチェック(not機械的な精査)」の指摘
   - 実際に5件のみを内容精査により移行、他60件は継続価値ありと判定

2. **内容精査の重要性**：
   - タイトル・メタデータのみでは適切な判定不可能
   - 文書の作成背景・現在の関連性・将来価値を総合的に検討する必要
   - 人間による文脈理解が必須

3. **新ルールの効果**：
   - 2週間経過：機械的判定で安全な自動アーカイブ
   - 2週間未満：内容精査による慎重な早期アーカイブ判定
   - 適切なバランスによる効率性と正確性の両立

### 7月実装

- patterns検索継続性確認
- 移行効果測定

### 継続運用
