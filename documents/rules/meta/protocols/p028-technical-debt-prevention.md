## 1. 概要

バグ修正・問題対応・技術的改善のすべてにおいて、根本解決を必須とし、やむを得ない一時対応には必ず返済計画を立てることで、技術的負債の蓄積を防ぎ、システムの長期的健全性を維持する。

## 2. 基本原則

### 2.1 対症療法の完全禁止
**問題**: 症状を隠すだけの場当たり的修正は必ず再発する  
**解決**: 根本原因の特定と除去を必須とする

### 2.2 技術的負債の累積防止
**問題**: 小さな妥協が積み重なり、大規模障害に発展  
**解決**: 負債の可視化と計画的返済

## 3. 禁止事項（違反は重大インシデント）

### 3.1 コード修正における禁止事項
- ❌ **対症療法的修正**: 症状が消えるが原因を特定していない修正
- ❌ **「とりあえず動くように」修正**: 一時的な回避コード
- ❌ **条件分岐による症状回避**: `if (問題状態) { 強制修正 }`パターン
- ❌ **段階的な最小調査**: 「少しずつログ追加」アプローチ

### 3.2 技術的判断における禁止事項
- ❌ **無計画な一時対応**: 返済計画なしの妥協
- ❌ **負債の隠蔽**: 問題を認識しながら記録しない
- ❌ **優先順位の誤り**: 新機能優先で負債を放置

## 4. 必須手順

### 4.1 問題対応フロー（例外なし実行）
1. **現象の完全記録**: 症状の詳細・再現条件・発生タイミング
2. **包括的調査システム実装**: 一度の実行で問題全容を把握（P020参照）
3. **複数原因仮説の列挙**: コードパス・データフロー・状態管理の各観点
4. **真因特定**: 調査データに基づく根本原因の確定
5. **解決方法の選択**:
   - 根本解決可能 → 即座に実装
   - 時間制約あり → 負債記録付き一時対応
6. **根本修正実装**: 原因を除去する修正（症状回避ではない）
7. **包括検証**: 同一原因による他の問題も解決されることを確認

### 4.2 技術的負債管理

#### 負債記録テンプレート
```javascript
// TECH_DEBT: [ID] [日付] [担当者]
// 問題: [具体的な問題の説明]
// 一時対応: [実施した応急処置]
// 根本解決案: [本来あるべき実装]
// 返済期限: [いつまでに解消するか]
// 影響範囲: [この負債が影響する機能]
// TODO: [根本解決時に実施すべきこと]
```

#### 返済計画の必須要素
- 具体的な実装方針
- 必要な工数見積もり
- 実施予定日（24時間以内に設定）
- 担当者の明確化

## 5. 違反検知チェック

### 5.1 修正前必須質問
1. 「この修正は症状を隠しているだけではないか？」
2. 「なぜこの症状が発生したかを特定できているか？」
3. 「同じ原因による他の問題も解決されるか？」
4. 「技術的負債を作っていないか？」
5. 「作るなら返済計画はあるか？」

**重要**: 「症状が消えた」≠「修正完了」

## 6. 緊急時の例外処理

### 6.1 一時対応が許容される条件（すべて満たす）
1. **本番障害**で即座の対応が必要
2. **根本解決に3時間以上**かかる見込み
3. **影響範囲が限定的**で他機能に波及しない
4. **24時間以内に返済計画**を立てる

### 6.2 負債の種類と影響度
| 種類 | 例 | 影響度 | 許容返済期限 |
|------|-----|--------|--------------|
| **コード負債** | 重複、ハードコード | 中 | 1週間 |
| **設計負債** | 不適切な構造 | 高 | 3日 |
| **ドキュメント負債** | 古い仕様書 | 低 | 2週間 |
| **テスト負債** | カバレッジ不足 | 高 | 3日 |
| **依存関係負債** | 古いライブラリ | 最高 | 即座 |

## 7. 成功指標

### 7.1 測定指標
1. **根本解決率**: 問題対応における根本解決の割合（目標: 90%以上）
2. **負債返済率**: 計画通りに返済された割合（目標: 80%以上）
3. **負債起因障害**: 技術的負債が原因の障害数（目標: 0件）
4. **平均返済期間**: 負債作成から解消までの日数（目標: 5日以内）

### 7.2 期待効果
- **即座の効果**: バグの再発防止、真因特定習慣、負債の可視化
- **長期的効果**: システムの安定性向上、開発速度の持続的向上、大規模障害の予防
