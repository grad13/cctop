# Status Directory - エージェント別プロジェクトステータス

このディレクトリには、5エージェント体制で各エージェントが個別に管理するプロジェクトステータスファイルが格納されています。

## ファイル構成

- `builder.md` - Builderエージェントの作業ログ（実装作業）
- `validator.md` - Validatorエージェントの作業ログ（検証・デプロイ）
- `architect.md` - Architectエージェントの作業ログ（設計・アーキテクチャ）
- `clerk.md` - Clerkエージェントの作業ログ（ドキュメント管理）
- `inspector.md` - Inspectorエージェントの作業ログ（監視・統計）

## 運用ルール

### 1. 記録の原則
- 各エージェントは自身のファイルにのみ記録する
- 他エージェントのファイルは読み取り専用
- 競合を避けるため、同時編集は行わない

### 2. DDD2階層メモリメンテナンス（P044準拠）
1. **L0→L1**: セッション終了前に各エージェントが自身のstatus/{agent}.mdを更新
2. **L1→L2**: 週次月曜日にP044に従って完了作業をrecords/reports/に移行
3. **role確認**: status更新後はroles/{agent}.mdで権限・責務を再確認
4. **進捗専用**: statusファイルは進捗記録専用（権限定義はrolesに分離）

### 3. 情報の粒度
- **即時記録**: 作業開始・完了・重要な決定
- **詳細情報**: 技術的な詳細、エラー内容、解決方法
- **次の作業**: 明確なタスクリスト形式

### 4. roles/status分離体制
- **権限・責務・制限**: roles/{agent}.mdで静的管理
- **作業進捗・現状**: status/{agent}.mdで動的管理（DDD2対象）
- **過去の履歴**: records/reports/で体系的管理（REP-XXXX形式）

## 利点
- エージェント間の編集競合を防止
- 各エージェントの作業が明確に分離
- 並行作業が安全に実施可能
- DDD2による体系的なメモリ管理
