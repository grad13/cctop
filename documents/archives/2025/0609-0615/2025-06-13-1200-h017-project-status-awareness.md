# H017: プロジェクト状況把握プロセス（統合版）

**仮説ID**: H017  
**作成日時**: 2025年6月13日 12:00  
**統合元**: H003 + H005  
**カテゴリ**: メタプロセス（統合）  
**ステータス**: 検証中  
**検証期間**: 2025年6月13日～6月20日

## 🎯 仮説文

**「セッション開始時の体系的状況把握プロセスにより、作業効率を40%向上させ、認識齟齬を90%削減できる」**

## 📋 背景（統合前の問題）

### H003から
- 仮説立案時の既存プロセス見落とし
- CLAUDE.md内の既存情報の後追い発見
- メタレベルでの作業品質管理不足

### H005から
- 「今何をやっているか」が即答できない
- セッション間での文脈喪失
- status/{agent}.mdの更新漏れ

## 🔬 統合ソリューション

### 1. セッション開始時必須プロセス（2分以内）

#### Step 1: 現状把握（30秒）
```
□ status/{agent}.md読了
  - 現在フェーズ確認
  - 前回終了時点確認
  - ブロッカー確認
```

#### Step 2: タスク確認（20秒）
```
□ TodoRead実行
  - アクティブタスク確認
  - 優先順位確認
```

#### Step 3: 関連情報スキャン（70秒）
```
□ 作業に応じた確認
  - 仮説作業 → hypotheses/README.md
  - 開発作業 → 該当specifications
  - ルール変更 → CLAUDE.md該当セクション
```

### 2. 作業中の状況維持

#### 30分ごとの確認
```
リマインダー：「現在の作業と次のステップは？」
- 現在のタスク進捗
- 次の具体的アクション
- ブロッカーの有無
```

#### 状態変化時の即時更新
- タスク完了 → TodoWrite + status/{agent}.md更新
- 新規発見 → 適切な文書に即座記録
- 問題発生 → ブロッカーとして記録

### 3. セッション終了時プロセス（1分）

```
□ status/{agent}.md更新
  - 完了項目追加
  - 次回開始点明記
  - 新規ブロッカー記載

□ 重要な発見の文書化
  - specifications/へ仕様追加
  - hypotheses/へ新規仮説
  - daily/へ作業記録
```

### 4. 3段階情報階層

#### Level 1: 即座把握（status/{agent}.md）
- 現在何をしているか
- 次に何をすべきか
- 何が問題か

#### Level 2: 詳細確認（各種README.md）
- 具体的な手順
- 関連情報
- 過去の経緯

#### Level 3: 深掘り（個別文書）
- 技術仕様
- 実装詳細
- 過去の議論

## 📏 検証方法

### 成功指標
1. **即答率**: 「現在の作業」を5秒以内に回答95%
2. **認識齟齬**: 10%以下（90%削減）
3. **作業効率**: 40%向上
4. **文脈保持**: セッション間90%以上

### 測定項目
- セッション開始時の把握時間
- 認識齟齬の発生数
- タスク完了時間
- status/{agent}.md更新頻度

## 💡 期待効果

### 即座の効果
- 作業開始の高速化
- 認識齟齬の激減
- 無駄な確認作業の削減

### 長期的効果
- プロジェクト全体の透明性向上
- 知識の体系的蓄積
- チーム拡張時の円滑な引き継ぎ

## 📝 実践ガイド

### 必須適用場面
- 毎セッション開始時
- コンテキストスイッチ時
- 長時間作業の中断再開時

### 軽量版（緊急時）
最低限30秒で：
1. status/{agent}.mdの現在フェーズのみ確認
2. TodoReadで最優先タスク確認
3. 作業開始
