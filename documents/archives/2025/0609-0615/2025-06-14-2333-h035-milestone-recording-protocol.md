# H035: 節目記録プロトコル（H025改訂版）

**作成日**: 2025年6月14日  
**ステータス**: 検証中  
**検証期間**: 2025年6月14日〜21日（1週間）  
**背景**: H025の3連続違反問題を受けた根本的プロトコル再設計

## 🎯 仮説

**「並行記録」は技術的に実装困難。エージェント特性に応じた「節目記録」方式により、実用的で遵守可能な記録プロトコルを確立できる。**

## 📋 問題分析

### H025失敗の根本原因
1. **「並行記録」の実装困難性**: 作業と記録の同時実行は認知負荷が高い
2. **エージェント特性無視**: コード作業とドキュメント作業の集中パターンの違い
3. **強制力の不足**: 違反しても実質的なペナルティがない

### 連続違反の事実
- **Monitor Agent**: 3連続違反（INC-20250614-017）
- **Coder Agent**: 3連続違反（INC-20250614-018）
- **共通パターン**: 作業集中時の記録失念

## 🔄 新プロトコル設計

### 基本コンセプト
**「並行記録」→「作業節目での確実記録」**

### エージェント別記録方式

#### 🔧 Coder Agent（強制型）
```
指示受信 → 「作業開始」記録 → コード作業 → 「重要進展」記録 → 「作業完了」記録
```

**記録タイミング**:
1. **開始記録**: 指示受信→即座に作業概要記録（必須）
2. **進展記録**: 重要な進展時（エラー解決、機能完成、方針変更）
3. **完了記録**: コミット前に必ず結果記録（必須・pre-commitフックで強制）

**強制メカニズム**:
- **pre-commitフック**: 30分以内にstatus更新がない場合コミットブロック
- **技術的強制力**: Gitレベルでの制御により回避不可能

**記録内容**:
- 1-2行の簡潔な状況説明
- 進捗状況、発生した問題、解決方法
- 次の作業計画

#### 📊 Monitor Agent（推奨型）
```
指示受信 → 「調査開始」記録 → 技術作業 → 「発見・問題」記録 → 「対応完了」記録
```

**記録タイミング**:
1. **開始記録**: 指示受信→作業概要記録（推奨）
2. **発見記録**: 重要な発見、問題、数値変化時
3. **完了記録**: 作業完了時の結果・統計

**運用方式**:
- **緩い運用**: 30分〜1時間での記録推奨
- **強制力なし**: 自主的な記録管理
- **内容重視**: 統計数値、発見事項、監視結果中心

#### 📝 Clerk Agent（現行維持）
```
文書作業では並行記録が適している
```

**継続理由**:
- 文書作業は記録と親和性が高い
- 実際に並行記録が可能
- 現行プロトコルで問題なし

## 🔧 技術的実装

### pre-commitフック
```bash
#!/bin/sh
# Coder Agent用 status更新強制チェック
# 30分以内の更新がない場合コミットブロック
```

**実装場所**: `.git/hooks/pre-commit`  
**対象**: Coder Agentのみ  
**強制力**: コミット自体をブロック

### 相互確認システム
- **作業時間制限**: 30分経過で記録チェック
- **他エージェント確認**: 長時間記録なしの場合に指摘

## 📊 検証方法

### 成功指標
1. **違反率**: 週間違反回数 0回（Coder）、2回以下（Monitor）
2. **実用性**: エージェントの主観的負担度 3以下（5段階）
3. **品質維持**: 記録内容の有用性・トレーサビリティ確保

### 検証期間中の観察項目
- 各エージェントの記録頻度・タイミング
- pre-commitフックによるブロック発生状況
- 記録内容の質・有用性
- エージェントの作業効率への影響

### 週次評価
- **毎週金曜**: 週間違反状況レビュー
- **主観評価**: 各エージェントからの使用感フィードバック
- **必要な調整**: タイミング、強制力レベル、記録内容の最適化

## 🎯 期待される効果

### 短期効果（1週間）
- H025違反の根絶（特にCoder）
- 実用的な記録習慣の確立
- ストレス軽減による作業効率向上

### 長期効果（1ヶ月〜）
- 自然な記録習慣の定着
- エージェント間の作業可視性向上
- プロジェクト全体のトレーサビリティ向上

## ⚠️ リスク・制約

### 技術的リスク
- pre-commitフックの誤作動
- 緊急時の回避手順整備必要
- 環境依存の動作差異

### 運用リスク
- 記録の形骸化
- Monitorの緩い運用による違反継続
- エージェント間の公平性問題
