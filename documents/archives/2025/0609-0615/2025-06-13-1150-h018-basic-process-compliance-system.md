# H018: 基本プロセス遵守システム（統合版）

**仮説ID**: H018  
**作成日時**: 2025年6月13日 11:50  
**統合元**: H004 + H006 + H009  
**カテゴリ**: プロセス改善（統合）  
**ステータス**: 検証中  
**検証期間**: 2025年6月13日～6月20日

## 🎯 仮説文

**「文書編集・開発作業における基本プロセスの遵守を体系化し、チェックリストと自動リマインダーにより、プロセス違反を95%削減できる」**

## 📋 背景（統合前の問題）

### H004から
- documentsディレクトリ編集時のREADME.md参照漏れ
- 構造破損・重複作成の頻発
- 関連文書の更新忘れ

### H006から  
- H004プロセス自体の違反発生
- 緊急時のプロセススキップ
- 認識齟齬対応時の基本プロセス軽視

### H009から
- LS/Grep実行忘れによる重複作成
- README.md/status確認スキップ
- 基本動作の形骸化

## 🔬 統合ソリューション

### 1. 3段階プロセス遵守システム

#### Level 1: 事前確認（必須）
```
□ status/{agent}.md確認（30秒）
□ 作業に応じた必須文書確認
  - documents編集 → README.md
  - 仮説関連 → hypotheses/README.md
  - 開発作業 → protocols/terminology.md
□ LS/Grepによる既存確認
```

#### Level 2: 作業中チェック（状況依存）
```
□ 命名規則遵守（file-naming-convention.md参照）
□ 関連文書のリストアップ
□ 更新影響範囲の特定
```

#### Level 3: 事後確認（必須）
```
□ README.md更新（構造変更時）
□ 関連文書の参照更新
□ daily記録への記載
```

### 2. 自動リマインダーシステム

#### トリガータイミング
- documents/へのWrite/Edit実行時
- 新規ファイル作成時
- ディレクトリ構造変更時
- 30分経過時の定期確認

#### リマインダー内容
```
「基本プロセス確認：
1. 事前確認は完了しましたか？
2. 関連文書は特定しましたか？
3. 更新が必要な文書はありますか？」
```

### 3. 違反防止メカニズム

#### 強制チェック
- 特定キーワード検出時の一時停止
  - 「新規作成」「ディレクトリ追加」「構造変更」
- 重要ファイル編集時の確認強化
  - README.md、status/{agent}.md、各種ドキュメント

#### エスカレーション
- 3回連続スキップで警告レベル上昇
- 重大違反時は作業中断・ユーザー確認

## 📏 検証方法

### 成功指標
1. **プロセス違反率**: 5%以下（95%削減）
2. **確認実行率**: 95%以上
3. **平均確認時間**: 3分以内
4. **構造破損**: 0件

### 測定項目
- 日次: 違反発生数、確認実行回数
- 週次: 違反率推移、時間効率
- 違反時: 原因分析、プロセス改善

## 💡 期待効果

### 即座の効果
- プロセス違反の劇的減少
- 構造破損・重複の防止
- 作業品質の安定化

### 長期的効果
- プロセス遵守の習慣化
- 認知負荷の軽減（自動化により）
- プロジェクト全体の品質向上

## 📝 運用ルール

### 適用範囲
- すべてのdocuments/編集作業
- 新規ファイル・ディレクトリ作成
- 大規模な構造変更
- 仮説・ルール関連作業

### 例外処理
- 緊急hotfix時も最低限Level 1は実施
- スキップ時は理由を明記
- 事後的にプロセス完了を確認

## 🔄 改善サイクル

1. **週次レビュー**: 違反パターンの分析
2. **プロセス最適化**: チェックリストの改善
3. **自動化推進**: 手動確認の自動化検討
