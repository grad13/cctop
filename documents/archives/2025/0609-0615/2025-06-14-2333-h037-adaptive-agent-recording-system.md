# H037: エージェント適応型記録システム（H025/H035/H036統合版）

**作成日**: 2025年6月14日 nodata  
**ステータス**: 実装完了・運用検証中  
**検証期間**: 2025年6月14日〜21日（1週間）  
**統合対象**: H025, H035, H036の完全統合版

## 🎯 統合仮説

**エージェントの作業特性に応じた記録システムにより、実用的で持続可能な記録プロトコルを確立できる。技術的強制力と自己監視システムの組み合わせで、記録忘れを根絶し、作業の可視性・トレーサビリティを向上させる。**

## 📋 統合経緯・背景

### H025失敗の教訓
- **並行記録の実装困難性**: 全エージェント統一は非現実的
- **3連続違反事例**: Monitor・Coder両方でLevel 2エスカレーション
- **エージェント特性無視**: 作業パターンの違いを考慮していない

### 統合による解決
- **H025**: 基本概念提供（置き換え完了）
- **H035**: Coder特化システム（技術的強制力）
- **H036**: Monitor特化システム（自己監視型）

## 🔧 エージェント別記録システム

### 💻 Coder Agent（強制型・H035実装）

#### 基本方式
```
指示受信 → 「作業開始」記録 → コード作業 → 「重要進展」記録 → 「完了記録」→ コミット
```

#### 技術的強制メカニズム
**pre-commitフック実装済み**:
```bash
#!/bin/sh
# 30分以内にstatus更新がない場合コミットブロック
if [ $DIFF -gt 1800 ]; then
    echo "🚫 COMMIT BLOCKED: Coder status未更新"
    exit 1
fi
```

#### 記録タイミング
1. **開始記録**: 指示受信→即座に作業概要記録（必須）
2. **進展記録**: 重要な進展時（エラー解決、機能完成、方針変更）
3. **バグ記録**: バグ発見・再現・解決時は`documents/bugs/`への記録（**新規義務化**）
4. **完了記録**: コミット前に必ず結果記録（pre-commitで強制）

#### 実装状況
- ✅ `.git/hooks/pre-commit`設置・実行権限設定
- ✅ macOS/Linux対応、詳細エラーメッセージ
- ✅ 技術的強制力により違反が物理的に不可能
- 🔄 **バグ記録システム追加実装予定**

### 📊 Monitor Agent（自己監視型・H036実装）

#### 基本方式
```
monitor/変更検知 → 作業フラグON → 5分後チェック → status未更新 → 警告・統計記録
```

#### 自己監視メカニズム
**monitor/self-watch.js実装済み**:
```javascript
// monitor/ディレクトリ変更を検知
chokidar.watch('monitor/').on('change', (path) => {
    monitorWorkDetected = true;
    setTimeout(checkStatusUpdate, 5 * 60 * 1000);
});
```

#### 記録タイミング
1. **開始記録**: 指示受信→作業概要記録（推奨）
2. **発見記録**: 重要な発見、問題、数値変化時
3. **完了記録**: 作業完了時の結果・統計

#### 実装状況
- ✅ `monitor/self-watch.js`作成・実行権限設定
- ✅ npm scripts統合（`npm run monitor-safe`）
- ✅ 統計ログ記録・記録遵守率自動計算
- ✅ 作業時のみリマインダー（アイドル時警告なし）

### 📝 Clerk Agent（従来型維持）

#### 基本方式
```
文書作業は並行記録が適している（従来のH025方式継続）
```

#### 継続理由
- 文書作業は記録と親和性が高い
- 実際に並行記録が可能
- 違反事例なし

## 🐛 Coderバグ記録義務化システム（追加実装）

### 対象範囲
**Coderが発見・対応するすべてのバグ**:
- 実装中のバグ発見
- ユーザー報告バグの調査・修正
- 既存コードの問題発見
- 表示・動作・パフォーマンス問題

### 記録必須タイミング
1. **バグ発見時**: 現象・再現手順・影響範囲を即座記録
2. **調査中**: 原因分析・仮説・検証結果を記録
3. **修正時**: 修正内容・テスト結果・副作用確認を記録
4. **完了時**: 最終的な解決内容・今後の予防策を記録

### 記録フォーマット
```markdown
# [YYYYMMDD] バグタイトル

**発見日時**: YYYY年MM月DD日 HH:MM  
**重要度**: 高/中/低  
**ステータス**: 発見/調査中/修正中/解決済み

## 📋 現象
- 何が起きているか
- 再現手順
- 影響範囲・頻度

## 🔍 原因分析
- 根本原因
- 関連コード箇所
- なぜ発生したか

## 🔧 修正内容
- 実装した修正
- 変更ファイル一覧
- テスト実施内容

## ✅ 確認事項
- 修正効果確認
- 副作用チェック
- 同種問題の予防策
```

### 技術的強制メカニズム
**pre-commitフック拡張**:
- バグ関連キーワード検出時に`documents/bugs/`記録確認
- 「bug」「エラー」「修正」等のコミットメッセージでバグ記録必須
- 記録なしの場合はコミットブロック

### 運用ルール
1. **即座記録**: バグ発見から30分以内
2. **1ファイル1バグ**: 1つのバグにつき必ず1つの専用ファイルを作成（**絶対制約**）
3. **継続更新**: 調査・修正の進展を随時更新
4. **完了確認**: 解決後は必ずステータス更新
5. **品質保証**: 詳細な記録で後続対応を効率化
