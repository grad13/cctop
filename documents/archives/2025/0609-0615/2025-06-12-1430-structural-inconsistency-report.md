# プロジェクト構造の重複・混乱に関する調査報告書

**作成日時**: 2025年6月12日 14:30  
**調査者**: Claude Code  
**目的**: プロジェクト内の構造的な重複・混乱を徹底的に調査し、改善策を提案する

## 🔴 主要な構造的問題

### 1. マイルストーン関連の重複

#### 現状
- **`/documents/roadmap/project-milestones.md`**
  - プロジェクト全体の実装ロードマップ
  - 7つのマイルストーンを定義（Timebox実装からマルチデバイス対応まで）
  - 各マイルストーンの概要と想定機能を記載
  
- **`/documents/milestones/`ディレクトリ**
  - 現在は`v1.0-unified-url-experience-complete.md`のみ
  - 完了したマイルストーンの詳細報告書

#### なぜ問題なのか
- 名称が紛らわしく、役割の違いが不明確
- `project-milestones.md`は計画、`milestones/`は完了報告という違いがあるが、直感的でない
- 新規参加者やClaude Codeが混乱する可能性

#### 推奨する解決策
```
documents/
├── roadmap/
│   ├── project-roadmap.md (旧 project-milestones.md を改名)
│   └── completed-milestones/ (旧 milestones/ を移動・改名)
│       └── v1.0-unified-url-experience.md
```

### 2. 命名規則の不統一

#### 現状の問題例
1. **日付フォーマットの混在**
   - `0610-timebox-ui-evaluation.md` (MMDD形式)
   - `2025-06-10-project-review.md` (YYYY-MM-DD形式)
   - `2025-04-27.md` (YYYY-MM-DD形式、説明なし)

2. **ファイル名の記述レベルの不統一**
   - 時刻あり: `2025-06-10-1330-virtual-desktop-implementation-review.md`
   - 時刻なし: `2025-06-10-project-review.md`
   - 日付のみ: `2025-06-10.md`

#### なぜ問題なのか
- 時系列での追跡が困難
- ファイルの目的が名前から判断しづらい
- CLAUDE.mdで定められた命名規則に従っていない

#### 推奨する解決策
- 一貫して`YYYY-MM-DD-HHMM-description.md`形式を使用
- 時刻記載が不要な場合は`YYYY-MM-DD-description.md`
- 日付のみのファイルは避け、必ず説明を含める

### 3. 用語管理の重複

#### 現状
- **`/documents/rules/glossary.md`** - プロジェクト用語集
- **`/documents/rules/terminology.md`** - 開発実装用語集  
- **`/documents/rules/naming-protocol.md`** - 用語管理ドキュメントの組織化ポリシー

#### なぜ問題なのか
- 3つのファイルで似た内容を扱っている
- どこに何を記載すべきか不明確
- 更新時に整合性を保つのが困難

#### 推奨する解決策
- `glossary.md`と`terminology.md`の役割は`naming-protocol.md`で明確化されているので、これに従う
- 重複箇所を特定して適切なファイルに統合

### 4. アーカイブ構造の非直感性

#### 現状
```
documents/archives/2025-06/
├── daily/
│   ├── 2025-04-27.md  # なぜ4月のファイルが6月フォルダに？
│   └── 2025-04-28.md
└── landing-page/
    └── (3つのファイル)
```

#### なぜ問題なのか
- アーカイブの年月と内容の年月が不一致
- アーカイブ基準が不明確（作成日？アーカイブ日？）

#### 推奨する解決策
- アーカイブフォルダ名を`archived-2025-06`のように変更
- アーカイブ日を明確にする
- または内容の年月でフォルダを分ける

### 5. 機能名称の変遷による混乱

#### 現状
- "Virtual Desktop" → "Quick Switch"への名称変更
- 関連ファイルに古い名称が残存
- seamless-navigation関連ファイルが複数バージョン存在

#### なぜ問題なのか
- 過去の経緯を知らない人には理解困難
- 検索時に古い名称では見つからない

#### 推奨する解決策
- 移行履歴を明確に文書化
- 古いファイルには新しい名称への参照を追加
