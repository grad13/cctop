# H043: ディレクトリ役割分担とMECE原則

**仮説ID**: H043  
**作成日**: 2025年6月15日 nodata  
**作成者**: Clerk Agent  
**状態**: 検証中  
**期間**: 2025年6月15日 - 2025年6月22日  
**カテゴリー**: 文書管理体系  

## 🎯 仮説

**記録系ディレクトリ（daily, status, incidents）では重複を許容し、それ以外のディレクトリではMECE（Mutually Exclusive, Collectively Exhaustive）原則を厳守することで、文書管理の柔軟性と体系性を両立できる。**

## 📋 背景

ユーザー指摘：
- dailyは各種記録を日時とともに極力保存する場所
- daily, status, incidentsでは重複が許される
- それ以外のディレクトリはMECEであってほしい

現状の課題：
- 同じ情報が複数箇所に散在し、どれが正式版か不明
- 更新時の同期漏れリスク
- 一方で、履歴的価値のある情報の保存も必要

## 🔬 提案する体系

### 1. ディレクトリの分類

#### A. 記録系ディレクトリ（重複許容）
履歴的価値を重視し、時系列での記録を優先

| ディレクトリ | 役割 | 重複方針 |
|-------------|------|----------|
| **daily/** | 日次の作業記録 | 完全許容（時系列優先） |
| **status/** | 各Agentの現在状況 | 部分許容（最新性優先） |
| **incidents/** | 問題発生記録 | 部分許容（追跡性優先） |
| **bugs/** | バグ記録・追跡 | 部分許容（状態遷移優先） |
| **archive/** | 過去文書の保存 | 完全許容（履歴保存） |
| **experiments/** | 実験・試行錯誤記録 | 完全許容（過程重視） |
| **reports/** | 実施記録・レポート | 部分許容（時系列保存） |

#### B. 体系系ディレクトリ（MECE原則）
情報の一意性と検索性を重視

| ディレクトリ | 役割 | MECE方針 |
|-------------|------|----------|
| **specifications/** | 仕様書 | 完全MECE（1機能1文書） |
| **hypotheses/** | 仮説管理 | 完全MECE（1仮説1文書） |
| **protocols/** | 手順書 | 完全MECE（1手順1文書） |
| **roadmaps/** | 計画書 | 完全MECE（1計画1文書） |
| **dominants/** | 最上位規則 | 完全MECE（重複厳禁） |
| **directions/** | 確立手法 | 完全MECE（1手法1文書） |

### 2. 重複管理の原則

#### 記録系での重複ルール
1. **daily**: すべての記録を時系列で保存（重複上等）
2. **status**: 現在の状態は1つ、過去はdailyへ移動
3. **incidents**: 1インシデント1ファイル、関連情報は参照で管理
4. **bugs**: active/archiveで状態管理、履歴は保持
5. **archive**: 削除より保存を優先、整理より蓄積
6. **experiments**: 失敗も成功も全て記録、過程を重視
7. **reports**: 実施記録を蓄積、傾向分析に活用

#### 体系系でのMECEルール
1. **単一責任**: 1文書1トピック
2. **相互排他**: 内容の重複禁止
3. **完全網羅**: カテゴリー内で漏れなし
4. **参照活用**: 関連情報は参照リンクで接続

### 3. 情報の流れ

```
作業発生
  ↓
status/{agent}.md に記録（現在進行）
  ↓
完了後、詳細をdaily/に切り出し（履歴保存）
  ↓
重要な成果は体系系ディレクトリへ（MECE配置）
  ↓
statusは要約のみ残して参照化
```

## 🎯 成功基準

### 定量的指標
1. **重複率**: 体系系ディレクトリでの内容重複 0%
2. **追跡性**: 過去1ヶ月の作業を100%追跡可能
3. **検索性**: 必要情報を3クリック以内で発見

### 定性的指標
1. **明確性**: どこに何があるか直感的に分かる
2. **保守性**: 更新箇所が明確で同期漏れなし
3. **柔軟性**: 新規情報の配置に迷わない
