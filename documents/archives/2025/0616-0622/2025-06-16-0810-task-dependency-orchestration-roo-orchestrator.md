## 1. 概要

タスク依存関係型オーケストレーションは、複雑なタスクを段階的に分解し、各段階で得られた知見を次段階に活用する「分割統治」アプローチ。Roo Orchestratorの設計思想をClaude Codeで実装する。

## 2. 核心原則

### 2.1 段階的知識構築
```
Step 1: 探索・分析
    ↓ (知見の圧縮・要約)
Step 2: 設計・計画
    ↓ (設計の具体化)
Step 3: 実装・実行
    ↓ (結果の検証)
Step 4: 統合・最適化
```

### 2.2 適応的計画修正
- 各ステップ完了後に次の計画を再評価
- 発見に基づく動的な調整
- 固定的な計画に縛られない柔軟性

## 3. Roo Orchestratorの特徴

### 3.1 コンテキスト圧縮
- **問題**: 全情報を保持するとコンテキストが肥大化
- **解決**: 各ステップで100-200語の要約を生成
- **効果**: コスト削減（$6→$1）、処理速度向上

### 3.2 並列と順次の組み合わせ
```yaml
orchestration:
  step_1:
    name: "全体分析"
    parallel_tasks:
      - analyze_codebase
      - analyze_requirements
      - analyze_constraints
    output: unified_analysis_summary
    
  step_2:
    name: "設計フェーズ"
    input: step_1.output
    parallel_tasks:
      - design_architecture
      - design_data_model
      - design_api
    output: integrated_design
```

## 4. Claude Code実装パターン

### 4.1 ステップ定義ファイル
```markdown
# orchestration-plan.md

## Step 1: 現状分析（30分）
### 並列タスク
1. **Task A**: 既存コードの構造分析
   - 入力: src/ディレクトリ
   - 出力: 構造サマリー（200語以内）

2. **Task B**: 依存関係の分析
   - 入力: package.json, requirements.txt
   - 出力: 依存関係マップ（200語以内）

3. **Task C**: 問題領域の特定
   - 入力: issues/, bugs/
   - 出力: 優先課題リスト（200語以内）

### 統合
- 3つのサマリーを統合して次ステップへ

## Step 2: 解決策設計（45分）
### 入力
- Step 1の統合サマリー

### 並列タスク
[以下省略]
```

### 4.2 実行コマンド
```markdown
# .claude/commands/orchestrate-complex.md

# orchestrate-complex

複雑なタスクをステップごとに分解して実行します。

## 使用方法
/project:orchestrate-complex <plan-file>

## 実行例
/project:orchestrate-complex "orchestration-plan.md"

## 処理フロー
1. 計画ファイルを読み込み
2. Step 1の並列タスクを実行
3. 結果を要約（各100-200語）
4. Step 2へ要約を引き継ぎ
5. 以降、全ステップ完了まで繰り返し
```

## 6. 効果測定

### 6.1 定量的効果
- **コスト削減**: 85%削減（$6→$1）
- **処理時間**: 60%短縮（並列実行による）
- **成功率**: 90%以上（段階的検証による）

### 6.2 定性的効果
- **透明性**: 各ステップの成果が明確
- **制御性**: 途中での方向修正が容易
- **再利用性**: ステップ単位での再実行可能

## 7. ベストプラクティス

### 7.1 ステップ設計
- **2-4ステップ**: 複雑すぎず、単純すぎない
- **明確な成果物**: 各ステップで具体的な出力
- **独立性**: ステップ内タスクは相互独立

### 7.2 要約戦略
- **構造化**: 箇条書きで要点整理
- **定量化**: 可能な限り数値で表現
- **次への示唆**: 次ステップへの推奨事項

### 7.3 エラー処理
- **ステップ単位**: 失敗したステップのみ再実行
- **部分的成功**: 成功したタスクの結果は保持
- **適応的修正**: エラーに基づく計画修正

## 8. 実装ロードマップ

### Phase 1: 基本実装（1週間）
- ステップ実行の基本フレームワーク
- 手動でのステップ定義
- 簡単な要約機能

### Phase 2: 自動化（2週間）
- 計画ファイルからの自動実行
- 動的な要約生成
- エラーハンドリング

### Phase 3: 高度化（1ヶ月）
- AIによる計画自動生成
- 適応的な計画修正
- 実行履歴からの学習

## 9. 他の並列化パターンとの使い分け

### 9.1 選択基準
| パターン | 適用場面 |
|---------|---------|
| 領域分割型 | 責任範囲が明確に分離可能 |
| データ並列型 | 同一処理の繰り返し |
| **依存関係型** | 段階的な知識構築が必要 |

### 9.2 組み合わせ利用
```
Orchestration (依存関係型)
└── Step 2
    ├── Task A: データ処理 (データ並列型)
    └── Task B: ドキュメント更新 (領域分割型)
```
