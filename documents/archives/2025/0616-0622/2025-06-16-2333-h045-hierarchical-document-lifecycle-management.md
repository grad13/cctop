# H045: 階層的文書ライフサイクル管理

**仮説ID**: H045  
**作成日**: 2025年6月16日 nodata  
**作成者**: Clerk Agent  
**カテゴリ**: 文書管理  
**ステータス**: 立案中  
**検証期間**: 2025年6月16日 - 2025年8月16日（2ヶ月）  

## 📋 概要

H044のCPUキャッシュメソッドに基づき、L2（reports）の鮮度を保つための階層的な文書ライフサイクル管理システムを確立する。月次レビューによる自動的な文書の昇格・降格により、各層の役割を明確に保つ。

## 🎯 背景と問題

### 現状の問題
1. **L2の汚染**: 古い計画書や完了済み文書がreportsに滞留
2. **手動依存**: 文書の移行が属人的判断に依存
3. **基準の欠如**: いつ・何を・どこに移すかが不明確
4. **メンテナンス不足**: 定期的な見直しプロセスがない

### 根本原因
- 文書のライフサイクルが定義されていない
- 各層（L1/L2/L3）の役割が実運用で曖昧化
- 移行プロセスが確立されていない

## 💡 仮説内容

### 中核となる仮説
**「月次の定期レビューによる階層的な文書管理により、各層の役割が明確に保たれ、必要な情報への高速アクセスが実現する」**

### 文書ライフサイクルの定義

```
誕生 → 成長 → 成熟 → 保存
L0   →  L1  →  L2  →  L3

L0: セッション中の作業（揮発性）
L1: 進行中の記録（日次更新）status/{agent}.md
L2: 完了した重要記録（週次〜月次）records/reports/
L3: 歴史的アーカイブ（年次）documents/archives/
```

### 移行基準の明確化

#### L1→L2移行基準
- 作業の完了（マイルストーン達成）
- 他エージェントとの共有必要性
- 2週間以上の安定性（頻繁な更新終了）

#### L2→L3移行基準
- **計画書**: 実施完了から1週間
- **作業記録**: 作成から3ヶ月経過
- **定期記録**: 更新停止から3ヶ月
- **例外**: 継続更新型（P007履歴等）は移行しない

### プロトコル体系

```
P024: 月次文書管理レビュー（トリガー）
  ├─ P023: L2キャッシュ移行プロトコル
  ├─ L1→L2昇格チェック
  ├─ 文書整合性確認
  └─ メトリクス収集
```

## 🔬 検証方法

### 成功指標
1. **L2鮮度**: 3ヶ月以内の文書が80%以上
2. **検索効率**: 必要文書の発見時間50%短縮
3. **移行精度**: 誤った移行ゼロ
4. **運用負荷**: 月次レビュー30分以内

### 測定項目
- 各層のファイル数推移
- 文書の平均滞留期間
- 移行作業の所要時間
- 誤移行の発生件数

### 検証スケジュール
- **Week 1-2**: プロトコル作成・初回実施
- **Month 1**: 初回月次レビュー実施
- **Month 2**: 改善点の反映・再実施
- **Month 2終了**: 効果測定・継続判断

## 🚨 リスクと対策

### 想定リスク
1. **過度な移行**: 必要な文書まで移行
2. **判断の複雑化**: 基準が複雑で運用困難
3. **レビュー負荷**: 月次作業が重すぎる

### 対策
1. **移行前確認**: チェックリストによる二重確認
2. **基準の簡素化**: 3つのシンプルな基準に集約
3. **自動化準備**: 将来的なスクリプト化を視野に
