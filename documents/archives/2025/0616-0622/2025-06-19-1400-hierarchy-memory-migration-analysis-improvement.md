# REP-0084: DDD2 L1→L2移行問題分析と改善提案

**作成日**: 2025年6月19日  
**作成者**: Clerk  
**タイプ**: 問題分析・改善提案  
**関連**: DDD2（階層メモリメンテナンス原則）、P043（L2→L3移行）

## 概要

エージェント性能低下の原因がstatus品質低下にあると推測し、DDD2のL1→L2移行メカニズムの問題を分析。成功時期（Commit:f1cd97b）との比較から、移行プロセスの明確化が必要と判明。

## 背景

### 観察された問題
1. **エージェント性能の低下** - 各エージェントの作業効率・判断精度が低下
2. **status品質の低下** - statusファイルの内容が希薄化している疑い
3. **成功時期の特定** - Commit:f1cd97b（2025年6月19日12:58）でのstatus内容がリッチ

### 仮説
status→reportsへの圧縮（L1→L2移行）による情報損失が、エージェントのコンテキスト理解を阻害している。

## 現状分析

### 1. DDD2の定義内容

#### 階層構造
- **L0**: セッション内作業記憶（揮発性）
- **L1**: status/{agent}.md（高速キャッシュ）
- **L2**: records/（中速キャッシュ）
- **L3**: archive/（長期保存）

#### 移行サイクル
- **日次**: L0→L1（セッション終了前）
- **週次**: L1→L2（完了作業の詳細化）
- **週次**: L2→L3（P043準拠）

### 2. L1→L2移行の現状

#### 定義されている内容（DDD2より）
```markdown
#### L1→L2移行時（必須確認）
- [ ] 作業は完了しているか
- [ ] 適切な配置先を選択したか（bugs/reports/incidents/）
- [ ] 1レポート1トピックを守っているか
- [ ] 参照URLを記載したか
- [ ] 疑問点・決定事項を記載したか
```

#### 問題点
1. **プロトコル化されていない** - チェックリストはあるが、具体的な手順書がない
2. **実施タイミングが曖昧** - 「週次」とあるが、いつ・誰が・どのように実施するか不明
3. **品質基準が未定義** - どの程度の詳細度でL2に記録すべきか不明

### 3. 成功時期のstatus分析（f1cd97b）

#### 特徴
- **適切な情報密度** - 必要十分な技術詳細を含む
- **明確な構造** - 完了・継続中・次の作業が整理されている
- **コンテキスト保持** - 作業の背景・理由が記載されている

#### 現在との差異
- 情報量自体は現在も十分だが、古い完了作業が残存している
- L1→L2移行が適切に実施されていない可能性

## 改善提案

### 1. L1→L2移行プロトコルの策定（新規P044）

#### 目的
- statusファイルの適切な情報密度維持
- エージェントパフォーマンスの向上
- 階層的メモリ管理の最適化

#### 内容案
1. **移行タイミング**
   - 毎週月曜日の最初のセッション
   - 各エージェント自身が実施（権限内）

2. **移行対象の選定基準**
   - 7日以上前の完了作業
   - 継続的な参照価値のある成果物
   - 技術的な学習・知見を含む内容

3. **L2記録の品質基準**
   - 背景・問題・解決策・成果の4要素を含む
   - 他エージェントが理解可能な自己完結性
   - 将来の類似問題への適用可能性

4. **L1残存内容の基準**
   - 現在進行中の作業
   - 1週間以内の完了作業（参照頻度高）
   - 継続的な監視・メンテナンス項目

### 2. 各エージェントstatus更新ガイドライン

#### 推奨構造
```markdown
## 🎯 現在の作業状況
### ⏳ 進行中
- 具体的な作業内容と進捗

### ✅ 直近完了（1週間以内）
- 完了日時と成果の要約
- 詳細はREP-XXXXを参照

### 🔄 継続的責務
- 定常的な監視・メンテナンス項目

## 📋 次の作業予定
- 優先度順のタスクリスト
```

### 3. 実施計画

1. **Phase 1**: P044プロトコル策定（Clerk実施）
2. **Phase 2**: 各エージェントへの通知（handoffs/経由）
3. **Phase 3**: 初回L1→L2移行の実施（各エージェント）
4. **Phase 4**: 効果測定・フィードバック収集

## 実施した追加分析（2025年6月19日）

### 役割混在問題の発見
statusファイルが以下の2つの役割を担っていることが判明：
1. **静的役割**: エージェントの権限・責務・制限の定義
2. **動的役割**: 作業進捗・現在状況の記録

この混在がDDD2の階層管理を困難にしている。

### 解決策の実装：B案採用
```
documents/
├── roles/              # 新設：権限定義専用（静的）
│   ├── builder.md
│   ├── validator.md
│   ├── architect.md
│   ├── clerk.md
│   └── inspector.md
├── status/             # 既存：進捗記録専用（動的・DDD2対象）
```
