# エージェントのフェーズ継続性に関する教訓

## 背景と発生した問題

### 具体的な問題事例
1. **情報の断絶**
   - Phase 1で設計したデータモデルがPhase 2で失われる
   - 意思決定の背景が伝わらず、矛盾した実装が発生
   - エージェント切り替え時にコンテキストが消失

2. **データ構造の後付け変更**
   - スキーマ変更がカスケード的に影響を及ぼす
   - マイグレーションコストが指数関数的に増大
   - データ整合性の確保が困難に

3. **引き継ぎの不完全性**
   - ドキュメントが散逸し、次のエージェントがゼロから再調査
   - 重要な制約事項や前提条件が伝わらない
   - 同じ問題に複数回直面

### 問題の根本原因分析
1. **システム的要因**
   - エージェントのLLMコンテキスト限界
   - セッション間での状態保持の困難さ
   - ファイルベースメモリの限界

2. **プロセス的要因**
   - フェーズ設計の不備
   - 引き継ぎプロトコルの未整備
   - ドキュメント管理の不徹底

3. **設計的要因**
   - データ設計の後回し
   - アーキテクチャ決定の遅延
   - スコープの不明確さ

## 核心的教訓
**agentはphaseを乗り越えられない可能性がある。最初にデータ周りは固めるべき**

## 詳細な理由分析

### 1. フェーズ間での情報断絶のメカニズム
- **コンテキスト切り替え**: 新しいセッションでは過去の詳細が失われる
- **メモリ限界**: 大量のドキュメントをすべて読み込めない
- **優先度判断**: 何が重要かの判断がエージェントごとに異なる

### 2. データ構造変更のコスト曲線
```
変更コスト = f(時間, 依存関係, データ量)

時間経過:     1週  →  1ヶ月  →  3ヶ月
変更コスト:     1x   →   5x    →   20x
リスク:       低   →   中    →   高
```

### 3. エージェントの認知限界
- **部分最適化**: 全体像が見えず、局所的な判断に陥る
- **コンテキスト依存**: 提供された情報のみで判断
- **学習の不連続性**: セッション間で学習がリセット

## 推奨アプローチの詳細

### 1. データ設計フェーズの確立

#### Phase 0: データ設計専用フェーズ
```mermaid
graph LR
    A[要件分析] --> B[エンティティ定義]
    B --> C[リレーション設計]
    C --> D[スキーマ定義]
    D --> E[マイグレーション計画]
```

#### 設計原則
1. **拡張性優先設計**
   - 将来の拡張ポイントを明示的に用意
   - JSONカラムやEAVパターンの活用
   - バージョニング機能の組み込み

2. **イミュータブルデータの分離**
   - コアデータとメタデータの明確な分離
   - イベントソーシングの採用検討
   - 監査ログの完備

3. **ドメイン境界の明確化**
   - マイクロサービスを意識した境界設計
   - APIファーストのアプローチ
   - データ所有権の明確化

### 2. エージェント継続性のメカニズム

#### 情報引き継ぎの3層構造
```
L1: コア情報（必須）
  - データモデル
  - APIインターフェース
  - ビジネスルール

L2: 実装詳細（重要）
  - 技術的決定事項
  - パフォーマンス考慮
  - セキュリティ要件

L3: 履歴情報（参考）
  - 議論の経緯
  - 却下された案
  - 実験結果
```

#### passage/handoffs/の活用方法
```bash
passage/handoffs/
├── templates/           # 引き継ぎテンプレート
│   ├── phase-transition.md
│   ├── agent-handoff.md
│   └── data-model-spec.md
├── pending/            # 待機中の引き継ぎ
│   ├── to-builder/
│   ├── to-validator/
│   └── to-architect/
└── completed/          # 完了した引き継ぎ
```

### 4. 具体的な実装例

#### データモデルドキュメントテンプレート
```yaml
# data-model-v1.0.yaml
version: 1.0
entities:
  User:
    fields:
      - name: id
        type: uuid
        constraints: [primary_key, not_null]
      - name: email
        type: varchar(255)
        constraints: [unique, not_null]
      - name: metadata
        type: jsonb
        description: "拡張用フィールド"
    
    indexes:
      - fields: [email]
        type: btree
      
    future_extensions:
      - "OAuth認証情報のmetadataへ格納予定"
      - "マルチテナント対応時にtenant_id追加"
```

#### 引き継ぎドキュメントテンプレート
```markdown
# Agent Handoff Document

## サマリー
- 引き継ぎ元: Architect Agent
- 引き継ぎ先: Builder Agent  
- 日時: 2025-06-23
- フェーズ: Design → Implementation

## コア情報
### データモデル
- 参照: documents/specs/data-model-v1.0.yaml
- 変更禁止フィールド: User.id, User.email
- 拡張可能フィールド: User.metadata

### 重要な制約
1. emailは大文字小文字を区別しない
2. 削除は論理削除のみ（deleted_at使用）
3. 個人情報は暗号化必須

## 次のアクション
1. マイグレーションスクリプト作成
2. Repositoryパターンでの実装
3. ユニットテスト作成
```

## 期待される効果

### 短期的効果（1-3ヶ月）
1. **開発速度の向上**
   - データ構造の再設計が不要
   - 意思決定の高速化
   - 手戻り作業の削減

2. **品質の安定化**
   - 一貫性のあるデータモデル
   - バグの早期発見
   - テストカバレッジの向上

### 長期的効果（6ヶ月以降）
1. **保守性の向上**
   - 技術的負債の減少
   - ドキュメントの充実
   - 新メンバーのオンボーディング短縮

2. **スケーラビリティ**
   - フェーズ分割の容易化
   - 並行開発の実現
   - マイクロサービス化への対応

## リスクと対策

### 識別されたリスク
1. **初期設計の過剰エンジニアリング**
   - 対策: MVPアプローチ、反復的改善

2. **フェーズ間のコミュニケーションオーバーヘッド**
   - 対策: テンプレート化、自動化ツール

3. **ドキュメントの陳腐化**
   - 対策: コードとの同期、CIでの検証

### 緩和策
1. **段階的導入**
   - 小規模プロジェクトでの試験
   - 成功体験の蓄積
   - ベストプラクティスの確立

2. **ツールサポート**
   - スキーマ管理ツールの導入
   - ドキュメント生成の自動化
   - 引き継ぎワークフローのシステム化
