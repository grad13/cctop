# Coderタスク分類・移行計画書

**作成日**: 2025年6月18日 06:25  
**作成者**: Clerk Agent  
**目的**: 既存Coderタスクを Builder/Validator へ適切に分類・移行する

## 📋 概要

現在Coder Agentが担当している作業を分析し、Builder Agent（実装担当）とValidator Agent（検証担当）への適切な分担を計画します。これにより、REP-0049の段階的移行を円滑に実施します。

## 🎯 Coder Agent の現在の責務（status/coder.md より）

### 主要責務
1. **ソースコードの実装・修正**
2. **バグ発見・分析・修正**（P037バグ記録義務付き）
3. **技術的問題解決**
4. **技術仕様書の実装詳細更新**
5. **コード品質の管理**
6. **バグファイルのactive/archive管理**

### 具体的な作業タイプ
- 新機能の実装
- 既存機能の改修
- バグ修正
- リファクタリング
- パフォーマンス改善
- テストコード作成
- 技術調査・検証
- コードレビュー（自己レビュー）

## 🔄 Builder/Validator への分類

### 🔨 Builder Agent へ移行するタスク

#### 1. 実装系タスク（100% Builder）
- **新機能実装**: 設計から実装まで
- **既存機能改修**: 機能追加・変更
- **バグ修正実装**: 問題の修正コード作成
- **リファクタリング**: コード構造改善
- **パフォーマンス改善**: 最適化実装

#### 2. 技術設計系タスク
- **実装レベルの技術設計**
- **アーキテクチャ決定**（実装範囲）
- **技術調査**: 実装方法の調査
- **プロトタイプ作成**

#### 3. 開発環境系タスク
- **開発ツール設定**
- **ビルド設定管理**
- **依存関係管理**（package.json等）

### ✅ Validator Agent へ移行するタスク

#### 1. 検証系タスク（100% Validator）
- **コード品質チェック**: 規約準拠確認
- **セキュリティ検証**: 脆弱性チェック
- **パフォーマンステスト**: 性能測定
- **統合テスト実行**
- **E2Eテスト実行**

#### 2. デプロイ系タスク
- **本番デプロイ実行**
- **デプロイ後確認**
- **ロールバック対応**
- **環境設定管理**

#### 3. 品質保証系タスク
- **コードレビュー**（第三者視点）
- **テスト網羅性確認**
- **リリース判定**
- **品質メトリクス測定**

### 🤝 Builder ↔ Validator 協働タスク

#### 1. バグ対応フロー
```
発見（Validator/ユーザー） → 分析（Builder） → 修正（Builder） → 検証（Validator）
```

#### 2. 新機能開発フロー
```
実装（Builder） → 初期テスト（Builder） → 品質検証（Validator） → デプロイ（Validator）
```

#### 3. 技術的問題解決
```
問題調査（Builder） → 解決策実装（Builder） → 効果検証（Validator）
```

## 📊 移行優先度

### 🔴 High（即座に移行）- パイロット運用で実施
1. **小規模バグ修正**: 明確な分担で練習に最適
2. **コード品質改善**: 低リスクで効果を確認しやすい
3. **単体機能追加**: 独立性が高く影響範囲限定

### 🟡 Medium（段階的移行）- 本格移行で実施
1. **中規模機能開発**: 複数ファイルに跨る実装
2. **パフォーマンス改善**: 計測・実装・検証の連携
3. **既存機能の大幅改修**: 影響範囲が広い

### 🟢 Low（後期移行）- 安定化後に実施
1. **アーキテクチャ変更**: 高リスク・高影響
2. **大規模リファクタリング**: 全体への影響大
3. **新技術導入**: 学習コスト高

## 🗓️ 段階的移行スケジュール

### Phase 1: パイロット運用（Day 4-5）
- **対象**: High優先度の小規模タスク
- **目標**: 基本的な連携フロー確立
- **想定タスク数**: 3-5個（TC-001, TC-002, TC-003）

### Phase 2: 本格移行（Day 6-7）
- **対象**: Medium優先度の実用的タスク
- **目標**: 実務レベルでの効率性確認
- **想定タスク数**: 5-10個

### Phase 3: 完全移行（Day 8-10）
- **対象**: 全タスクタイプ
- **目標**: Coder役割の完全移行
- **残タスク**: Low優先度は必要に応じて

## ⚠️ 移行時の注意事項

### 1. 責任境界の明確化
- **Builder**: 「動くコード」の責任
- **Validator**: 「品質保証」の責任
- **グレーゾーン**: 事前に協議・ルール化

### 2. 情報伝達の確保
- **実装意図**: Builderが明確に文書化
- **品質基準**: Validatorが事前に共有
- **フィードバック**: 具体的・建設的に

### 3. 効率性の維持
- **過度な分割回避**: 小さすぎるタスクは非効率
- **バッチ処理**: 類似タスクはまとめて依頼
- **並行作業**: 独立タスクは並行実施
