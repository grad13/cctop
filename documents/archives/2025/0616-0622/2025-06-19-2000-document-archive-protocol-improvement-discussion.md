# REP-0087: records/サブディレクトリ設計議論とアーカイブプロトコル改善

**作成日**: 2025年6月19日  
**作成者**: Clerk Agent  
**カテゴリ**: 設計討議・プロトコル改善  
**関連**: P043（L2→L3移行）、P044（L1→L2移行）、DDD2（階層メモリメンテナンス）

## 📋 概要

REP files内容精査によるDDD2 L2→L3アーカイブ適用後、P043・P044プロトコルの改善とrecords/サブディレクトリ設計について包括的に議論。機械的判定の問題点、内容精査の重要性、README.md肥大化問題、RAGによる根本解決の可能性まで検討した。

## 🚨 発端：機械的アーカイブの問題

### 問題の発生
- **状況**: REP-0020〜0086の65件を機械的に移行しようとした
- **ユーザー指摘**: 「機械的に調べましたよね ダメです、ちゃんと読んでください」
- **実際の結果**: 内容精査により5件のみアーカイブ、残り60件は継続価値ありと判定

### 教訓
- タイトル・メタデータのみでは適切な判定不可能
- 文書の作成背景・現在の関連性・将来価値を総合的に検討する必要
- 人間による文脈理解が必須

## 🔧 P043プロトコル改善

### 改善前の問題
- **6ヶ月経過**: 期間が長すぎて実用的でない
- **機械的判定**: 内容を考慮しない自動判定

### 改善後のルール
1. **2週間経過ルール（機械的判定）**
   - 作成から2週間（14日）経過 → 自動的にアーカイブ対象
   - 判断不要・機械的に移行実行

2. **2週間未満ルール（内容精査必須）**
   - 作成から2週間未満でも内容精査により早期アーカイブ可能
   - **禁止**: 機械的・自動的な判定
   - **必須**: 人間による内容確認・判断

### 内容精査プロセス
- **ファイル全文読了**: タイトルやメタデータのみでなく、内容全体を確認
- **作成背景理解**: なぜ作成されたか、何の問題を解決したかを理解
- **現在の関連性**: 現在進行中のプロジェクト・タスクとの関連を確認
- **将来の参照価値**: 今後参照される可能性を検討
- **代替情報源**: 同じ内容が他の文書で体系化されているか確認

## 🔄 P044プロトコル改善

### 改善内容
1. **3日間ルール導入**: 経験則に基づく適切な期間設定
2. **容量基準追加**: status 500行超過時の強制実施
3. **内容理解による分類**:
   - **bugs**: DDD0の最も抽象レベルの低い問題（具体的なコード・機能の不具合）
   - **incidents**: DDD0のmeta以上の層での違反（プロセス・ルール・システム違反）
   - **reports**: その他全て（実装・分析・設計・調査・完了記録等）

### 分類判定プロセス
1. 具体的なバグ・不具合か？ → records/bugs/
2. メタレベル・プロセス違反か？ → records/incidents/
3. その他・判断に迷う場合 → records/reports/（デフォルト）

## 🤔 records/サブディレクトリ設計の検討

### 現状の問題認識
- **reports/の内容**: 計画書、設計書、進捗記録、草案、分析、完了報告など多様
- **名称の不整合**: 「reports」という名前が実態と合わない
- **分類の曖昧さ**: 多様な内容をreportsに一括収容

### 検討されたサブカテゴリ案

#### 案1: 内容別詳細分類
```
records/
├── bugs/
├── incidents/
├── plans/         # 計画書・設計書・提案書
├── progress/      # 進捗記録・作業ログ・実施記録
├── drafts/        # 草案・原理原則・分析
└── reports/       # 完了報告・調査結果・サマリー
```

#### 案2: シンプル分類
```
records/
├── bugs/
├── incidents/
├── documents/     # 計画・設計・草案・原理等
└── activities/    # 進捗・作業記録・実施記録
```

### 詳細分類を見送った理由

#### 1. 分類判断の複雑化
- P044の内容精査時間が増加
- 境界ケースでの判断迷い
- Agent負荷の増大

#### 2. REP番号体系との整合性
- 既存のREP-0020〜0086の再分類が必要
- 参照整合性の維持コスト

#### 3. 将来の構造変更コスト
- 「構造を変更することになった場合、それはそれで再度面倒になる」
- ディレクトリ構造変更の連鎖的影響
- Agent間での新ルール周知コスト

## 📊 README.md肥大化問題

### 問題の発見
- **現状**: REP-0020〜0086で約67件、既に相当な行数
- **予想**: 年内に100-150件、README.md全体が500行超える可能性
- **影響**: Agentがrecords/確認時のcontext圧迫

### 解決案の検討

#### 1. サブカテゴリによるREADME分散
- 各カテゴリで20-30件程度に分散
- context負荷の軽減
- **問題**: 構造変更コストとのトレードオフ

#### 2. README軽量化アプローチ
- 最新10-20件のみ表示
- 月別README分割
- 動的生成によるREADME自動作成

### activeファイル数の自然上限分析

**現状の傾向**:
- 実質的なactive件数は約62件（67件から5件アーカイブ済み）

**activeファイルの自然減要因**:
1. P043による定期archive（2週間経過で自動移行）
2. プロジェクト完了による価値減少
3. 体系化・統合による自然淘汰

**予想される上限**:
- 進行中プロジェクト関連: 10-20件
- 最近の実装記録: 20-30件
- 未完了の計画・分析: 10-20件
- **合計: 40-70件程度で安定の可能性**

**結論**: P043・P044が適切に機能すれば、activeファイル数は自然に適正範囲に収まる可能性

## 🔍 RAGによる根本解決の可能性

### RAG (Retrieval-Augmented Generation) アプローチ

#### 現在の問題
- Agent が documents/ 全体を context に載せる必要
- README.md 肥大化による context 圧迫
- 全文書を毎回読み込む非効率性

#### RAG による解決
- **検索ベース**: 必要な文書のみを動的に取得
- **関連性判定**: クエリに関連する文書のみを context に追加
- **段階的読み込み**: まず索引→関連文書特定→詳細読み込み

### REP-0079との関連

**REP-0079（MCP自律化アーキテクチャ設計）**で既に検討済み:
- **patterns検索の活用**: キーワードベースでの関連文書発見
- **段階的情報取得**: 全体把握→詳細確認の2段階アプローチ
- **context効率化**: 必要最小限の情報のみをAgentに提供

### RAG導入の制約
- **Claude Max**: RAG機能が使用不可
- **ローカル環境**: 実装・運用が面倒
- **現実的判断**: いったん延期

### 現在のアプローチの位置づけ
- **多量のタグによるRAG代替**: アーカイブ時の5-10キーワード付与
- **patterns検索継続性**: archive/ 内でのキーワード検索による発見
- **理想的ではないが実用的**: 現在の技術制約下での合理的解決

## 🎯 最終決定事項

### 1. records/サブディレクトリ構造
**現状維持**: bugs/incidents/reports の3分類を継続
- **理由**: 分類判断が簡単、構造変更コストなし、archive後は検索ベースのため分類精度は重要でない

### 2. プロトコル改善
- **P043**: 2週間 + 内容精査ルールで確定
- **P044**: 3日間 + 内容精査ルールで確定

### 3. README.md肥大化対策
**様子見アプローチ**: activeファイル数の自然上限を検証
- P043・P044の適切な運用により40-70件程度で安定する可能性
- 実際に問題が顕在化してから対策検討

### 4. 将来のRAG検討
- 技術制約が解決されれば根本的な解決手段として有効
- 現在は多量タグ + patterns検索で代替

## 💡 重要な洞察

### 1. 機械的判定の限界
- タイトル・メタデータのみでは文書の真の価値を判定不可能
- 文脈理解と人間の判断が不可欠

### 2. アーカイブの本質
- records/ での分類は一時的な整理に過ぎない
- archive/ でのフラット格納 + patterns検索が最終形態
- L2での分類精度よりも、L3での検索継続性が重要

### 3. スケーラビリティ問題の段階的解決
- README.md肥大化は将来的な問題
- activeファイル数の自然上限が存在する可能性
- 問題が顕在化する前に過度な対策は不要

### 4. RAGの位置づけ
- context管理問題の根本的解決手段
- 現在の多量タグアプローチはRAGの簡易代替
- 技術制約解決時の移行パスとして重要

## 🔄 今後の方針

1. **短期**: P043・P044の改善ルールで運用開始
2. **中期**: activeファイル数とREADME.mdサイズの動向監視
3. **長期**: RAG技術制約の解決とアーキテクチャ移行検討

---

**結論**: 理想的ではないが、現在の技術制約と運用効率を考慮した現実的な解決策として、現状維持 + プロトコル改善で対応する。
