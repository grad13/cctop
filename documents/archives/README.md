## 🔄 DDD2 L2→L3移行プロトコル（P043準拠）

### 移行判断基準
1. **6ヶ月経過**: records/での作成から6ヶ月経過
2. **3ヶ月アクセス0**: 直近3ヶ月間のアクセス実績なし
3. **完了プロジェクト**: 関連プロジェクト・機能の完了
4. **容量制限**: records/の容量制限到達時

### 移行実施手順
1. **移行対象選定**: P043判断基準による候補抽出
2. **キーワード抽出**: 5-10個の検索キーワード付与
3. **週次ディレクトリ配置**: archives/YYYY/MMDD-MMDD/形式
4. **patterns検索確保**: 将来の検索継続性確認
5. **README.md更新**: アーカイブ内容の記録

### 実施済みアーカイブ（2025年6月）
- **REP-0001〜0012**: 初期プロジェクトレポート（19件、2025年6月19日追加移行）
  - 第4回追加: REP-0001（meta整理）、REP-0002（meta整理v2）、REP-0003（P007履歴）、REP-0039（hypotheses統合）、REP-0041（プロトコル再配置）
- **bugs/**: TimeBox/TaskGrid関連バグ（9件）
- **daily/**: プロジェクトレビュー作業記録
- **hypotheses/**: 統合済み仮説（H025等）
- **surveillance-implementations/**: SQLite移行実装記録

## 🎯 L3長期保存の特徴

### アクセス頻度最適化
- **低頻度アクセス**: 月1回未満の参照向け
- **検索ベース**: patterns/grepでの検索が主要アクセス方法
- **コンテキスト保持**: README.mdによる背景・関連情報維持

### キーワード戦略
- **5-10個付与**: 過少・過多を避けた適切な粒度
- **技術用語優先**: 検索しやすい技術キーワード
- **プロジェクト名**: 関連機能・システム名の明記
- **時期情報**: 開発フェーズ・リリース時期

## 📊 週次アーカイブ形式

### ディレクトリ命名規則
- **形式**: `YYYY/MMDD-MMDD/`
- **開始**: 月曜日開始（ISO週番号準拠）
- **例**: `2025/0616-0622/`（2025年6月16日〜22日）

### README.md必須項目
```markdown
# Archive YYYY/MMDD-MMDD

**アーカイブ日**: YYYY年MM月DD日
**対象期間**: records/での活動期間
**移行理由**: P043判断基準の適用結果
**キーワード**: [5-10個のキーワード]

## 概要
アーカイブ内容の簡潔な説明

## ファイル一覧
- REP-XXXX: タイトル（キーワード）
- BUG-XXXX: タイトル（キーワード）

## 検索継続性
patterns検索での発見方法・関連キーワード
```
