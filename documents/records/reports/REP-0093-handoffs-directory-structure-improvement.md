# REP-0093: Handoffsディレクトリ構造改善提案

**作成日**: 2025年6月24日  
**作成者**: Clerk Agent  
**カテゴリ**: システム改善提案  
**優先度**: Medium  
**ステータス**: 提案  

## 概要

passage/handoffs/のディレクトリ構造を、より一貫性のある3段階ワークフロー構造に改善する提案。

## 現状の問題点

### 現在の構造
```
handoffs/
├── pending/to-{agent}/     # 処理待ち（統一的）
├── in-progress/{agent}/    # 処理中（統一的）
├── {agent}/                # 完了タスク（エージェントごとに分散）
├── user/outbox/            # ユーザーからの指示
└── shared/                 # 共有リソース
```

### 問題点
1. **不整合な構造**: pending/とin-progress/は統一的だが、完了タスクは各エージェントディレクトリに分散
2. **管理の複雑化**: 完了タスクを確認するには複数のディレクトリを巡回する必要
3. **アーカイブの困難**: 各エージェントディレクトリから個別にアーカイブ作業が必要

## 改善提案

### 提案する新構造
```
handoffs/
├── pending/
│   ├── to-builder/
│   ├── to-validator/
│   ├── to-clerk/
│   └── to-inspector/
├── in-progress/
│   ├── builder/
│   ├── validator/
│   ├── clerk/
│   └── inspector/
├── completed/              # 新設：全完了タスクを統一管理
│   ├── 2025-06-24/        # 日付別に整理
│   │   ├── builder/
│   │   ├── validator/
│   │   └── ...
│   └── README.md
├── user/outbox/
├── shared/
└── archive/               # 既存：長期保存
```

### 変更内容
1. **completed/ディレクトリの新設**
   - 全エージェントの完了タスクを一元管理
   - 日付別サブディレクトリで整理
   - エージェント別サブディレクトリで分類

2. **各エージェントディレクトリの廃止**
   - builder/, validator/, clerk/, inspector/を削除
   - 完了タスクはcompleted/へ統一

## 実装計画

### Phase 1: 準備（30分）
1. completed/ディレクトリ構造の作成
2. completed/README.mdの作成
3. 移行スクリプトの準備

### Phase 2: 移行（1時間）
1. 既存の各エージェントディレクトリから完了タスクを抽出
2. completed/の適切な日付ディレクトリへ移動
3. 空になったエージェントディレクトリを削除

### Phase 3: ドキュメント更新（1時間）
1. passage/handoffs/README.mdの更新
2. shared/quick-start-guide.mdの更新
3. 各エージェントrole文書のhandoffs参照更新

## 期待効果

### 1. 管理の簡素化
- **統一的な3段階フロー**: pending → in-progress → completed
- **完了タスクの一覧性**: completed/で全体を俯瞰可能
- **検索の効率化**: 日付・エージェント別に整理

### 2. アーカイブの効率化
- **バッチ処理可能**: completed/から一括でarchive/へ
- **日付管理**: 古い日付ディレクトリを丸ごとアーカイブ
- **保存期間の統一**: 全エージェント共通のポリシー適用

### 3. 新規エージェントの追加容易性
- **ディレクトリ作成不要**: pending/in-progress/completedに追加するのみ
- **統一的なワークフロー**: 全エージェント同一の処理フロー

## リスクと対策

### リスク
1. **移行中の混乱**: 作業中タスクの所在が不明になる可能性
2. **参照の破損**: 既存ドキュメントのパス参照エラー

### 対策
1. **段階的移行**: まず新規タスクから新構造を適用
2. **完全なドキュメント更新**: 全参照箇所の事前調査と一括更新

## 代替案

### 代替案1: 現状維持
- **メリット**: 変更不要、既存の慣れ
- **デメリット**: 不整合な構造の継続

### 代替案2: inbox/outbox構造への回帰
- **メリット**: 当初設計との整合性
- **デメリット**: pending/in-progressの明確性を失う

## 推奨事項

1. **本提案の採用**: completed/ディレクトリによる統一管理
2. **段階的実装**: 新規タスクから適用開始
3. **ドキュメント先行**: README等を先に更新して周知

## 結論

completed/ディレクトリの導入により、handoffsシステムは真に統一的な3段階ワークフローを実現できる。これは管理効率とシステムの一貫性を大幅に向上させる。
