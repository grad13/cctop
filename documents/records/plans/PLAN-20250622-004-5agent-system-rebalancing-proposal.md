# 5 Agent体制リバランス提案 - src/test/spec等価重要性の実現

**Document ID**: PLAN-20250622-004  
**Created**: 2025-06-22 23:15  
**Author**: Validator Agent  
**Status**: Proposal  
**Purpose**: cctop v3の教訓とRDD経験を踏まえた5 Agent体制の再編成提案

## 🎯 提案の背景

### cctop v3から得られた重要な教訓

1. **テスト成功率100%でも実行時クラッシュ**: モックと実装の乖離による品質問題
2. **RDD（実動作駆動開発）の重要性**: `npm start`での動作確認がテストより重要
3. **設計・実装・検証の不均衡**: Builder/Validatorが実装寄り、Architectが軽視される傾向

### 現在の5 Agent体制の問題点

```
現在の状況:
Builder + Validator = 実装系（coder寄り）
Architect = 設計系（孤立気味）
Clerk = 文書系
Inspector = 監視系

問題:
❌ src重視、spec軽視
❌ Architectの地位が不明確
❌ test品質がモック依存で実動作軽視
❌ 専門性の重複（Builder/Validator両方が実装関与）
```

## 📊 核心原則：等価重要性の確立

### **src, test, spec の3要素は等価に重要**

```
src/  ← Builder Agent専門
test/ ← Validator Agent専門  
spec/ ← Architect Agent専門
```

この原則により、開発の三本柱（設計・実装・検証）が均等に重視される体制を構築する。

## 🔄 新しい5 Agent体制提案

### **Builder Agent** - src専門
- **専門領域**: `src/` ディレクトリの実装
- **責務**: 
  - コード実装・機能開発・技術実装
  - 動作するコードの確実な実装
  - Architectからの設計仕様の正確な実装
- **品質基準**: 
  - コードが仕様通りに動作する
  - 保守性・可読性の高い実装
  - パフォーマンス要件の達成

### **Validator Agent** - test専門（RDD対応）
- **専門領域**: `test/` ディレクトリの品質検証とRDD実践
- **責務**: 
  - **実動作確認**（最優先）
  - 統合テスト・E2Eテスト実行
  - モックと実装の乖離検出
  - 継続的な動作監視
- **品質基準**: 
  - `npm start`での確実な動作
  - 実ユーザー環境での動作保証
  - エンドツーエンドでの品質保証

### **Architect Agent** - spec専門（地位向上）
- **専門領域**: 仕様書・設計文書・アーキテクチャ
- **責務**: 
  - システム設計・仕様策定・技術方針決定
  - アーキテクチャの一貫性保証
  - 技術的意思決定の最終責任
- **品質基準**: 
  - 一貫性のある設計・仕様
  - 実装可能で現実的な設計
  - 長期保守性を考慮した設計

### **Clerk Agent** - 文書系（継続）
- **専門領域**: CLAUDE.md・プロセス・メタ文書
- **責務**: 
  - ドキュメント管理・ルール策定
  - プロセス改善・文書整合性管理
- **品質基準**: 
  - 文書の整合性・アクセス性
  - プロセスの実用性

### **Inspector Agent** - 監視系（継続）
- **専門領域**: surveillance/・統計・監視
- **責務**: 
  - システム監視・データ分析・運用監視
  - 長期的な品質トレンド分析
- **品質基準**: 
  - 継続的な動作監視
  - 有用な統計・分析情報の提供

## 🎯 重要な変更点

### 1. **Architect Agent の地位向上**

#### 現在の問題
- 設計が軽視され、「たまに相談される」程度の存在
- 技術方針決定の権限が不明確
- 仕様書作成が後回しにされる

#### 提案する改善
- **全システムの設計責任者**として明確に位置づけ
- Builder/Validatorと対等な権限・責任を付与
- 技術的意思決定の最終責任者として機能

### 2. **Validator Agent のRDD特化**

#### 現在の問題
- テスト実行中心で実動作確認が軽視
- モックテストに依存し、実装との乖離を見逃す
- Builder Agentと実装領域で重複

#### 提案する改善
- **実動作確認を最優先**とするRDD実践者
- モックと実装の乖離検出専門
- 実ユーザー視点での品質保証

### 3. **Builder Agent の実装特化**

#### 現在の問題
- Validator Agentと責務が重複
- 品質検証まで担当し、専門性が分散

#### 提案する改善
- **実装のみに特化**し、高い専門性を発揮
- Architectからの設計を正確に実装
- 動作するコードの実装に集中

## 📋 権限・責務の再配分

### **新しい権限マトリクス**

| Agent | 主要ディレクトリ | 副次的権限 | 協調範囲 |
|-------|-----------------|------------|----------|
| Builder | `src/` | 実装関連documents/ | Architect→Builder |
| Validator | `test/` | 品質関連documents/ | Builder→Validator |
| Architect | `spec/` | 設計関連documents/ | Validator→Architect |
| Clerk | documents/rules/ | CLAUDE.md | 全Agent文書管理 |
| Inspector | surveillance/ | records/reports/ | 全Agent監視・分析 |

### **連携フロー強化**

```
Architect (設計) → Builder (実装) → Validator (検証)
     ↑                                    ↓
     ←────── フィードバックループ ←──────
```

#### **1. Architect → Builder**
- 設計仕様の明確な受け渡し
- 実装方針・技術選択の指示
- アーキテクチャ制約の伝達

#### **2. Builder → Validator**  
- 実装完了の品質検証依頼
- 実装詳細の説明・引き継ぎ
- 既知の制限事項・注意点の共有

#### **3. Validator → Architect**
- 品質課題・設計問題のフィードバック
- 実動作での発見事項の報告
- 改善提案・設計見直し依頼
