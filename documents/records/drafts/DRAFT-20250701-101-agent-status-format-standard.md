# DRAFT-20250701-101: Agent Status Format Standard

**作成日**: 2025年7月1日  
**作成者**: Clerk Agent  
**カテゴリ**: 設定・プロトコル更新関連  
**状態**: 新規作成（Builder status実装例を基に規格化）

## 📋 概要

Agent statusファイルの一貫したフォーマット規格を定義し、DDD2（階層メモリメンテナンス原則）のL1キャッシュとしての機能を最適化する。

## 🎯 背景・問題

### 現状の課題
1. **フォーマット不統一**: Agent間でstatus記録形式がバラバラ
2. **情報散在**: 引き継ぎ・KPT・完了事項の配置が不規則
3. **追記混乱**: 時系列が不明確で情報検索が困難
4. **L1機能不全**: 「即座作業継続」のためのキャッシュ機能が低下

### 解決目標
- **一貫性確保**: 全Agent共通の統一フォーマット
- **情報整理**: 4つの要点（日時・引き継ぎ・KPT・追記）の体系化
- **検索性向上**: 時系列による情報アクセスの高速化
- **L1最適化**: DDD2におけるL1キャッシュ機能の強化

## 📐 Agent Status Format Standard

### 🗂️ **基本構造**

```markdown
# [Agent Name] Agent Status

【STOP】ここで一旦停止 → 先に `documents/agents/roles/[agent].md` を読んでください
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
**⛔ 更新禁止**: この注意書きの変更・削除は絶対禁止です。

**最終更新**: YYYY-MM-DD HH:MM JST  
**現在作業**: [現在の作業状況を一行で記述]

## 🎯 現在の作業状況（YYYY-MM-DD HH:MM）

### **引き継ぎ資料**
[現在進行中・次の作業に必要な情報]

### **Problem & Keep & Try**
[現在の改善点・継続事項・挑戦事項]

---

## 🎯 以前の作業状況（YYYY-MM-DD HH:MM）

### **引き継ぎ資料**
[該当時点での引き継ぎ情報]

### **Problem & Keep & Try**
[該当時点でのKPT]

---

## ✅ 完了事項
[日付別の完了実績記録]
```

### 📋 **必須4要素**

#### 1. **日時記入原則**
- **最終更新**: ファイル冒頭に必須（`YYYY-MM-DD HH:MM JST`形式）
- **セクション日時**: 各作業状況セクションに明記
- **追記時刻**: 新規追記時は必ず時刻を記録

#### 2. **引き継ぎ記入原則**
- **現在進行中**: 未完了作業・継続タスク
- **次の推奨作業**: 優先度付きの作業リスト
- **技術課題**: 解決待ちの問題・ブロッカー
- **handoffs状況**: 他Agentとの連携事項

#### 3. **KPT記載原則**
- **🔴 Problem（改善事項）**: 最大3項目、具体例必須
- **🟢 Keep（継続事項）**: 最大3項目、評価・強化点明記
- **🔵 Try（挑戦事項）**: 最大3項目、取り組み・目標設定

#### 4. **追記形式原則**
- **時系列順**: 新しい情報が上部、古い情報が下部
- **区切り線**: `---`で各追記セクションを明確に分離
- **完了事項分離**: 底部の独立セクションで実績管理

### 🔄 **更新頻度・タイミング**

#### **必須更新タイミング**
1. **セッション開始時**: 現状確認・作業計画の更新
2. **作業完了時**: 実施内容・成果・次の作業の記録
3. **重要決定時**: 方針変更・技術選択の記録
4. **handoffs処理時**: 他Agentとの連携記録

#### **推奨更新頻度**
- **30分毎**: 長時間作業での中間記録
- **コミット前**: Git操作前の状況記録
- **問題発生時**: トラブル・解決策の即座記録

### 📊 **品質管理原則**

#### **記録品質要件**
1. **具体性**: 抽象的表現禁止、行番号・ファイル名等の具体例必須
2. **簡潔性**: 冗長説明回避、必要最小限の情報記載
3. **検索性**: キーワード・参照リンクの適切な配置
4. **一貫性**: 用語・表記の統一

#### **肥大化防止（P044準拠）**
- **基準**: 300行超過時にP044プロトコル強制実行
- **L1→L2移行**: 詳細をreports/に切り出し
- **要約保持**: 必要最小限の情報のみstatus保持

## 📝 実装例（Builder Agent）

### **良い例**
```markdown
### **引き継ぎ資料**

#### **v0.3.0 最小限実装完了**

1. **実装状況**
   - **モジュール構造**: modules/shared, modules/daemon, modules/cli の3モジュール構成確立
   - **動作確認済み**: Daemonのファイル監視とDB記録が正常動作

2. **既知の問題**
   - **CLIの表示**: getRecentEventsは実装したが、実際のイベント表示が未実装
   - **HO-20250630-001**: システムクラッシュ問題が未対応

### **Problem & Keep & Try**

#### 🔴 **Problem（改善事項）**

1. **worktree統合の複雑さ**
   - 具体例: 複数worktreeの異なる実装を統合しようとして失敗、最小限実装に方針転換
   - 指摘: 複雑な統合より、動作する最小限から始めるべきだった
```

### **悪い例**
```markdown
### 引き継ぎ

作業完了しました。

### KPT

問題があった。良かった点もある。次回気をつける。
```

## ⚠️ 注意事項・制限

### **絶対禁止事項**
1. **注意書き変更**: 冒頭の【STOP】〜更新禁止注意書きの変更・削除
2. **時系列破綻**: 新しい情報を下部に追加（上部は最新維持）
3. **KPT省略**: Problem・Keep・Try のいずれかの省略
4. **抽象的記録**: 具体例なしの曖昧な記述

### **推奨回避事項**
1. **過度な詳細**: reports/で記録すべき詳細をstatusに記載
2. **重複記録**: 同じ内容の複数箇所記録
3. **長期保持**: 完了から3日経過した詳細情報の保持

## 🎯 成功指標

### **定量指標**
- **作業開始時間**: セッション開始から実作業開始まで5分以内
- **情報検索時間**: 過去情報検索を30秒以内
- **更新頻度**: 1セッションあたり最低2回のstatus更新

### **定性指標**
- **一貫性**: 全Agent statusの形式統一達成
- **可読性**: 新規参加者でも5分以内でstatus理解可能
- **保守性**: status肥大化の自動防止機能
