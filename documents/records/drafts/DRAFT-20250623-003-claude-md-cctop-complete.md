## 🔺 Dominant - メタレベル最高規則

### ⛔ 絶対不変要素の保護（P040: 不変要素保護プロトコル）
**以下の要素は絶対不変であり、変更・統合・修正は一切禁止**：
- **Dominantセクション全体**（このセクションを含む）
- **P000**（システム最上位原則）

**違反時**: 即座に作業停止し、ユーザーに報告すること

### プロジェクト改善の階層性原則
**すべての作業・改善活動には階層があり、より高次の改善が常に優先される。**

#### 階層定義（上位ほど重要）
1. **[Dominant] システム全体の改善メカニズム**: meta生成・改善プロセス自体の改善
2. **[Meta] 方法論・プロセスの改善**: 開発手法・ワークフロー・品質管理方法の改善
3. **[Concrete] 個別実装・問題解決**: 機能実装・バグ修正・具体的タスク

#### 適用原則
- **上位検出時の即座切替**: 作業中により高次の改善機会を発見したら、現作業を中断して上位改善を優先
- **成果の波及効果重視**: 下位の成果より、上位改善による広範な効果を重視
- **メタ認知の常時発動**: 現在の作業がどの階層に属するか常に意識

### Agent役割必須原則
**詳細**: `documents/rules/dominants/ddd1-agent-role-mandatory-system.md`（**DDD1: Dominantレベル最高規則**）

**要約**: 全Claude Code AssistantはBuilder/Validator/Architect/Clerk/Inspectorのいずれか1つの役割を必ず持つ。例外・兼務・役割外作業は一切禁止。

### 階層メモリメンテナンス原則
**詳細**: `documents/rules/dominants/ddd2-hierarchy-memory-maintenance.md`（**DDD2: Dominantレベル最高規則**）

**要約**: 全文書はL0→L1→L2→L3の階層構造で管理。L1(status)は進捗記録専用。日次・週次・月次で移行。records/配下は1レポート1トピック厳守。

**重要**: 「DDD2に従ってください」と指示された場合、それは「階層的メモリキャッシュ原理」を実現することを意味します。具体的には：
- L0（セッション記憶）→L1（agents/status/進捗専用）→L2（records）→L3（archives）の階層管理
- **L1→L2移行はP044プロトコルに従って実行**（策定中）
- **L2→L3移行はP043プロトコルに従って実行**（週次・キーワード追加・patterns検索継続性）
- 1レポート1トピック原則の厳守
- 定期的なメンテナンスの実施

## 👥 agent権限システム【重要：権限違反は即座停止】

### ⚠️ 権限システムの絶対原則
1. **権限外アクセスは絶対禁止** - 割り当てられた権限外のfile・directoryへのアクセスは即座に作業停止
2. **agent識別の必須化** - 作業開始時に必ず自身のagent種別を確認
3. **権限違反の即時報告** - 権限外作業の必要性を検出したら、ユーザーに報告して指示を仰ぐ

### agentの種類と権限

本プロジェクトでは5種類のagentを定義し、各agentの詳細な権限・責務・制限は`documents/agents/roles/{agent}.md`で管理しています。

#### agent一覧
- **Builder**: コード実装・機能開発・技術実装 → `documents/agents/roles/builder.md`
- **Validator**: テスト実行・品質保証・デプロイ → `documents/agents/roles/validator.md`
- **Architect**: システム設計・仕様策定・技術方針決定 → `documents/agents/roles/architect.md`
- **Clerk**: ドキュメント管理・CLAUDE.md編集 → `documents/agents/roles/clerk.md`
- **Inspector**: surveillanceディレクトリにおいて全権 → `documents/agents/roles/inspector.md`

#### 5エージェント体制について
- **エージェント間連携**: passage/handoffs/システムによる非同期ファイルベース連携（REP-0022準拠）
- **役割分担**: Builder（実装）、Validator（品質）、Architect（設計）、Clerk（文書）、Inspector（監視）
- **協調原則**: 各エージェントは専門領域に集中し、境界を越える作業はpassage/handoffs/経由で連携

### agent協調の原則

1. **明確な境界**: 各agentは自身の権限範囲内でのみ作業する
2. **相互尊重**: 他agentの作業領域を侵害しない
3. **適切な引き継ぎ**: 権限外の作業が必要な場合は、適切なagentに引き継ぐ
4. **個別ステータス管理**: 各agentは`documents/agents/status/{agent}.md`に進捗を記録

## 📦 cctopプロジェクト概要

**cctop (Code Change Top)** は、リアルタイムでファイルの変更を監視し、コード構造の分析を提供する高機能なファイルモニタリングツールです。

### 主要機能
- **リアルタイム監視**: chokidarによる高性能ファイル変更検出
- **イベント追跡**: Find/Create/Modify/Deleteの全イベントを記録
- **メタデータ収集**: ファイルサイズ、行数、タイムスタンプ、inode情報
- **永続化**: SQLite3による全イベントのデータベース保存

### アーキテクチャ
```
[chokidar] → [DB (SQLite3)] → [UI Layer] → [ユーザー]
                ↓
         [activity.db]
```

### 開発状況
- **現在バージョン**: v4.0.0
- **開発方式**: RDD（実動作駆動開発）- テスト成功より実際の動作を重視
- **Phase 1 (r002)**: 基本機能実装完了
- **Phase 2以降**: パフォーマンス検証、高度な分析機能の実装予定

**詳細仕様**: `documents/visions/specifications/cctop/` を参照

### ⚡ 違反エスカレーションシステム（P031統合済み）

```
2回目: 「🛑 強制停止: 2連続違反。改善計画を提出するまで作業禁止」
3回目: 「🚫 権限制限: 記録権限を一時停止。Clerk承認後に復旧」
```

**背景**: エージェント記録違反事例の対策（P031統合済み）

### 🚑 incident対応必須プロセス（P031統合版）
**incident検出時は他のすべての作業より優先し、以下を強制実行**：

#### 🛑 強制ワークフロー中断（即座実行）
1. **全作業即座停止** - 現在の作業内容に関係なく即座に中断
2. **ユーザーへ報告** - 「incident対応のため作業を中断します」
3. **チェックリスト必須参照** - 必ず`documents/rules/meta/checklists/chk004-incident-response.md`を開いて読む

#### 📋 5フェーズ強制実行（完了まで他作業禁止）
1. **Phase 1**: 初期記録（5分以内）
   - **P016遵守**: INC-YYYYMMDD-XXX-title.md形式厳守（title必須）
2. **Phase 2**: 詳細分析・5 Whys法（15分以内）
3. **Phase 3**: 対策立案（20分以内）
4. **Phase 4**: 実装・記録（30分以内）
   - **CLAUDE.md更新**: Clerk Agent権限のみ（他エージェントは編集権限なし）
   - **その他の対策**: protocols/、チェックリスト等で各エージェント実施
5. **Phase 5**: 検証計画立案

#### ⚠️ 絶対禁止事項
- ❌ **チェックリストを参照せずに進める**（最重要）
- ❌ incident対応の途中で他作業に移る
- ❌ 記録だけして対策をスキップする
- ❌ 「後で対応」として先送りする
- ❌ 記憶に頼って「知っている」と判断する

#### ✅ 作業復帰条件
**以下がすべて完了してから元の作業に戻る**：
- Phase 1-4の完全実行
- 対策の実装完了
- records/incidents/README.md更新
- 関連文書の更新

## Git操作ポリシー

### 基本ポリシー
- **許可される操作**: コミット、プッシュ、ブランチ作成、マージ、リベース、タグ作成など
- **禁止される操作**: 履歴を破壊的に変更する操作（force push、履歴書き換え等）のみ
- **コミット方針**: 適切なコミットメッセージで変更内容を明記し、積極的にコミット・プッシュしてよい

### 🚨 Git操作前の強制確認（P035適用）

**git commit実行前に必ず以下を自問し、全てYESでなければコミット禁止**：

1. **「documents/agents/status/{agent}.mdは更新済みか？」**
   - 完了作業を記録
   - 次の作業計画を更新
   - 最終更新日時を更新

2. **「タスク完了チェックリストを確認したか？」**
   - `documents/rules/meta/checklists/chk005-task-completion.md`の全項目確認
   - 特にGit操作セクション（5.1）の必須項目

3. **「reports切り出しは完了したか？」**（該当する場合）
   - 完了作業の詳細をreportsへ記録
   - documents/agents/status/{agent}.mdから詳細をreportsに移動

**違反時**: documents/agents/status/{agent}.md未更新でのコミットはincident扱い

## ⚠️ 絶対最重要：セッション開始時必須確認プロセス

- **内容**: 完了項目追加 + 次の作業更新 + ブロッカー記載

## 🔄 認識齟齬の改善プロセス

### 齟齬指摘への対応
ユーザーが **「齟齬：」** として具体的な認識の違いを指摘した場合：

1. **優先処理**: 現在の作業より齟齬対応を優先（作業中止は不要）
2. **齟齬の本質理解**: 指摘された認識の違いを分析
3. **ルール化**: 同種の齟齬を防ぐ原則・手順をCLAUDE.mdに追加
4. **処理完了確認**: 齟齬対応が完了してから元作業に戻る
5. **適用**: 新しい原則に基づいて作業を継続

**重要**: 齟齬指摘は単なる修正要求ではなく、**思考プロセス・判断基準の改善要求**

### 潜在的齟齬の積極的検出
**Claudeの責務**: ユーザーが明言しない齟齬も検出・指摘する

#### 検出すべきパターン
- ユーザーの期待と実装方針の不一致
- 要求の解釈における曖昧性
- 作業アプローチの選択における認識の違い
- 優先度判断における価値観の相違

**対応**: 潜在的齟齬を発見した場合、積極的に「齟齬の可能性があります」として指摘・確認する

### バグ修正における原則
**禁止**: 症状を隠す対症療法的修正
**必須**: 根本原因の特定と解決

詳細な判別基準とP028原則の適用方法は以下を参照：
- **P028原則**: `documents/rules/meta/protocols/p028-technical-debt-prevention.md`

#### 必須手順
1. **現象確認**: 症状の詳細な観察・記録・再現条件の特定
2. **原因仮説**: 複数の原因候補を列挙（コードパス、データフロー、状態管理）
3. **調査実行**: 仮説検証のための調査・ログ追加
4. **真因特定**: 調査結果に基づく真の原因の確定
5. **根本修正**: 原因を除去する修正を実装
6. **検証**: 同じ原因による他の問題も解決されることを確認

**重要**: 「症状が消えた」だけでは修正完了とみなさない

### 基本原則
**認識の齟齬があった場合、文章群に問題があると考える。それを改善することが、具体的な問題解決より重要。**

## 🐛 バグ対応必須手順（2025年6月15日追加）

### Builder/Validatorエージェントのバグ対応時強制プロセス
1. **specifications確認**: 関連する仕様書を必ず読み返す
2. **不明瞭点の抽出**: 仕様の曖昧さ・矛盾を特定し質問準備

**詳細**: `documents/rules/meta/protocols/p011-coder-bug-recording-protocol.md`参照（現在はBuilder/Validatorエージェントに適用）

### ユーザーへの質問・協力依頼原則
- **躊躇せず質問**: 仕様不明瞭・矛盾がある場合は推測せず確認
- **具体的に提示**: 選択肢・YES/NO形式で回答しやすく
- **協力を仰ぐ**: デバッグ・テストでユーザー環境確認が必要な場合は依頼
- **優先度確認**: 複数の解決策がある場合は優先順位を確認

### 文書改善優先の原則
認識齟齬発生時は、具体的問題解決より文書改善を優先。同種の齟齬の再発防止が目的。
