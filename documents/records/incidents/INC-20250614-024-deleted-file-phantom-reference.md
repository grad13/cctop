# インシデント記録: 削除ファイルの幻影参照問題

**インシデントID**: INC-20250614-010  
**発生日時**: 2025年6月14日 14:15  
**報告者**: ユーザー  
**対応者**: Claude Inspector  

## 現象

LEGACY_STATUS.mdファイルを削除した後も、file-mapping.jsonにファイルID:6として`"documents/LEGACY_STATUS.md"`のエントリが残存し、監視システムが削除されたファイルを参照し続けている。

## 重要度

**High** - 監視システムの整合性とデータ品質に関わる問題

## 影響範囲

- 監視システムのファイル管理の整合性破綻
- 削除ファイルの統計データが継続的に記録される可能性
- APIレスポンスに存在しないファイルの情報が含まれる
- データ分析時の誤った情報提供

## 詳細分析

### 技術的な問題
1. **ファイル削除検知**: `chokidar.watch()`の`unlink`イベントは発生
2. **記録処理**: `recordChange(filePath, 'DELETE')`は実行され、lines:0で記録
3. **マッピング残存**: `file-mapping.json`からのエントリ削除は未実装
4. **ID再利用**: 削除ファイルのIDが新規ファイルに再割り当てされない

### 監視ログの確認
```
📝 記録: documents/LEGACY_STATUS.md (ID:6, 0行) [DELETE]
```
- DELETEイベントは正常に記録されている
- しかしfile-mapping.jsonには残存

## 5 Whys分析

### Why1: なぜLEGACY_STATUS.mdのエントリが残存しているか？
→ ファイル削除時にfile-mapping.jsonからエントリを削除する処理が実装されていない

### Why2: なぜエントリ削除処理が実装されていないか？
→ 監視システム設計時に「ファイル削除＝マッピング削除」の要件が考慮されていない

### Why3: なぜその要件が考慮されていないか？
→ ファイルIDの永続性とマッピングの整合性管理の設計が不十分

### Why4: なぜ設計が不十分だったか？
→ ファイルのライフサイクル全体（作成→変更→削除）を通した整合性管理の検討不足

### Why5（根本原因): なぜライフサイクル全体の検討が不足したか？
→ 監視システム開発時に削除イベントを「記録するだけ」と捉え、システム状態の整合性維持を軽視した

## 即時対応

1. **手動マッピング修正**: file-mapping.jsonからLEGACY_STATUS.mdエントリを削除
2. **削除処理改善**: ファイル削除時のマッピング自動削除機能実装
3. **整合性チェック**: 他の削除ファイルの残存確認

## 再発防止策

1. **ファイル削除プロトコル強化**
   - DELETE時の自動マッピング削除
   - ID再利用防止メカニズム

2. **整合性検証機能**
   - 定期的なマッピング-ファイル実在性チェック
   - 不整合検出時の自動修正

3. **テスト強化**
   - ファイル削除シナリオの網羅的テスト
   - マッピング整合性の継続的検証

## 対策実装

- [ ] ファイル削除時のマッピング自動削除機能実装

## 学習事項

**監視システムでは「記録」だけでなく「システム状態の整合性維持」が重要。** ファイル削除は単なるイベントではなく、システム全体の状態変更として扱う必要がある。

## 技術的詳細

### 現在のfile-mapping.jsonの問題箇所
```json
{
  "lastId": 120,
  "files": {
    "6": "documents/LEGACY_STATUS.md",  // ← 削除済みファイルが残存
    ...
  }
}
```

### 修正すべき処理フロー
1. `unlink`イベント検知
2. `recordChange()`でDELETE記録
3. **追加**: `removeFileMapping(fileId)`でマッピング削除
4. **追加**: IDプール管理で再利用制御
